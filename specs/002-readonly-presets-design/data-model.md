# Data Model — Internal Catalog Resolution & ProviderConfig Scoping

**Feature**: `002-readonly-presets-design` | **Plan**: [plan.md](./plan.md)

**Date**: 2026-05-31 (respec; supersedes the 2026-05-20 catalog-as-CRDs data model)

This document enumerates every Kubernetes API entity this feature adds, refactors, or removes, and the one internal (non-Kubernetes) data structure introduced — the resolver cache. Each entity is described in terms of operator-facing surface (`spec`, validation), controller-managed surface (`status`), and lifecycle relationships. Field types are Go-style for precision; CRD YAML is generated by `controller-gen` and shipped under `package/crds/`.

The MVP entities (`Project`, `SshKey`, `ContainerRegistry`'s repository, etc.) carried forward unchanged are NOT re-described here — see [`001-mvp-scaffolding/data-model.md`](../001-mvp-scaffolding/data-model.md).

---

## 1. Kubernetes entities

### 1.1 `ProviderConfig` (namespaced) — NEW

**API**: `timeweb.crossplane.io/v1alpha1`, kind `ProviderConfig`, **scope: Namespaced**.

**Purpose**: Crossplane v2 namespaced credential reference. Referenced by namespaced MRs in the same namespace. Per spec FR-001.

**Spec**:

```go
type ProviderConfigSpec struct {
    Credentials ProviderCredentials `json:"credentials"`
}

type ProviderCredentials struct {
    Source xpv1.CredentialsSource `json:"source"`                  // typically "Secret"
    CommonCredentialSelectors `json:",inline"`                     // SecretRef { name, namespace, key }
}
```

**Status**: standard `xpv1.ProviderConfigStatus` (carries `users` counter via `ProviderConfigUsage`).

**Validation**: `credentials.source == "Secret"` enforced via kubebuilder enum; `secretRef.name` required.

**Relationships**:

- Referenced by MRs in the same namespace via `spec.providerConfigRef: { name: <pc-name> }`.
- Owns `ProviderConfigUsage` (namespaced) objects via the standard runtime finalizer pattern.

**Lifecycle**: standard Crossplane; deletion is blocked while any `ProviderConfigUsage` references it.

---

### 1.2 `ClusterProviderConfig` — NEW (renamed from MVP `ProviderConfig`)

**API**: `timeweb.crossplane.io/v1alpha1`, kind `ClusterProviderConfig`, **scope: Cluster**.

**Purpose**: The cluster-scoped sibling. Resolved when an MR's `spec.providerConfigRef.kind` is `ClusterProviderConfig` (also the crossplane-runtime v2 default when `kind` is omitted). Per the 2026-05-31 upstream-alignment clarifications (spec.md §Clarifications), there is **no silent fallback** from a missing namespaced `ProviderConfig` to a same-named `ClusterProviderConfig`; the operator chooses by setting `kind` explicitly.

**Spec**: shares the single `ProviderConfigSpec` shape with the namespaced `ProviderConfig` (§1.1) — both embed `xpv1.CommonCredentialSelectors`. For this kind the secret's `namespace` field is required (a cluster-scoped CR cannot infer a namespace); the namespaced PC defaults it to its own namespace at controller-lookup time.

**Status**: standard `xpv1.ProviderConfigStatus`.

**Relationships**:

- Owns `ClusterProviderConfigUsage` (cluster-scoped) objects.

**Migration from MVP**: today's cluster-scoped MVP `ProviderConfig` is renamed to `ClusterProviderConfig`. The MVP CRD is replaced; operators reapply existing PC manifests with the new kind. Per the spec's Assumptions, no external consumers exist on `main`; no conversion webhook is shipped.

---

### 1.3 `ProviderConfigUsage` / `ClusterProviderConfigUsage` — REFACTORED

**API**: `timeweb.crossplane.io/v1alpha1`. The MVP's single `ProviderConfigUsage` (cluster-scoped) is split into:

- `ProviderConfigUsage` (Namespaced) — tracks usage from same-namespace `ProviderConfig` references.
- `ClusterProviderConfigUsage` (Cluster) — tracks usage from `ClusterProviderConfig` references.

Both are managed entirely by `crossplane-runtime`; the provider does not author finalizer logic.

---

### 1.4 `ContainerRegistry` — REFACTORED

**API**: `containerregistry.m.timeweb.crossplane.io/v1alpha1`, kind `ContainerRegistry`, **scope: Namespaced**.

**Purpose**: Manage a Timeweb container registry. Refactored to the new XOR sizing shape per FR-004.

**Spec**:

```go
type ContainerRegistryParameters struct {
    // Exactly one of:
    PresetName *string                                  `json:"presetName,omitempty"`
    Resources  *ContainerRegistryResourcesParameters    `json:"resources,omitempty"`

    // Other operator-facing fields carried over from MVP:
    // (e.g. Name, ProjectRef — unchanged)
}

type ContainerRegistryResourcesParameters struct {
    DiskGB int64 `json:"diskGB"`                    // bucket disk capacity (free-form integer)
    // Container Registry is platform-managed: no operator-visible hypervisor /
    // location placement, no CPU/RAM/GPU input. Disk size is the only custom
    // dimension. Per spec.md §Clarifications 2026-05-31 (catalog-endpoint
    // reality check), there is no configurator endpoint for registries.
}

// +kubebuilder:validation:XValidation:rule="(has(self.presetName) ? 1 : 0) + (has(self.resources) ? 1 : 0) == 1",message="exactly one of presetName or resources must be set"
type ContainerRegistrySpec struct {
    xpv2.ManagedResourceSpec `json:",inline"`
    ForProvider              ContainerRegistryParameters `json:"forProvider"`
}
```

**Status**:

```go
type ContainerRegistryObservation struct {
    // Locked sizing discriminator (mirrors which side of the XOR was used at first create):
    LockedPresetID  *int64                            `json:"lockedPresetID,omitempty"`
    LockedResources *ContainerRegistryLockedResources `json:"lockedResources,omitempty"`

    // … other observation fields carried from MVP (UpstreamID, URL, etc.)
}

type ContainerRegistryLockedResources struct {
    DiskGB int64 `json:"diskGB"`
}
```

Exactly one of `lockedPresetID` (when the operator used the `presetName` variant) or `lockedResources` (when they used the `resources` variant) is populated after first successful create. The discriminator is consulted on every subsequent reconcile per FR-009/FR-010. There is no `lockedConfiguratorID` for Container Registry — the field exists on MR kinds that DO have a configurator (Server, K8s).

**Conditions** (extending the standard `Ready`/`Synced`):

- `Synced=False, reason=PresetNotFound` — operator's `presetName` did not resolve.
- `Synced=False, reason=PresetAmbiguous` — operator's `presetName` matched multiple upstream entries.
- `Synced=False, reason=SizingSwitchRequiresRecreate` — operator switched between `presetName` and `resources` on an existing MR.
- `Synced=False, reason=CatalogUnauthorized` — token denied a catalog endpoint.
- (`NoConfiguratorAvailable` is N/A for Container Registry — kept for Server / K8s MRs.)

**Relationships**:

- References `ProviderConfig` or `ClusterProviderConfig` via `spec.providerConfigRef.{kind,name}`; the controller hard-switches on `kind` with no silent fallback per FR-001 (post upstream-alignment clarification).
- May reference a `Project` (cross-MR reference, unchanged from MVP).

**Lifecycle**: standard Crossplane managed-resource lifecycle. The locked sizing variant is set on first successful Create and is immutable afterwards (a `presetName`↔`resources` switch surfaces as `SizingSwitchRequiresRecreate`; the operator must delete and recreate).

### 1.5 `S3Bucket` — REFACTORED

**API**: `objectstorage.m.timeweb.crossplane.io/v1alpha1`, kind `S3Bucket`, **scope: Namespaced**.

**Purpose**: Manage a Timeweb S3-compatible bucket. Refactored to the new XOR sizing shape.

**Spec**:

```go
type S3BucketParameters struct {
    PresetName *string                          `json:"presetName,omitempty"`
    Resources  *S3BucketResourcesParameters     `json:"resources,omitempty"`

    // Other fields (e.g. Name, ProjectRef, StorageClass, Type) — unchanged
    // from MVP and live OUTSIDE the XOR sizing block.
}

type S3BucketResourcesParameters struct {
    DiskMB int64 `json:"diskMB"`                 // bucket capacity (free-form integer; upstream API uses MB)
    // S3 Storage is platform-managed: no operator-visible location/hypervisor
    // placement, no configurator endpoint upstream (spec.md §Clarifications
    // 2026-05-31 catalog-endpoint reality check). Disk size is the only
    // custom dimension. `storageClass` ("hot" | "cold") is a separate field
    // outside the XOR block — it's an MR-level enum validated by static CEL,
    // not a sizing input.
}

// +kubebuilder:validation:XValidation:rule="(has(self.presetName) ? 1 : 0) + (has(self.resources) ? 1 : 0) == 1",message="exactly one of presetName or resources must be set"
type S3BucketSpec struct { … }
```

**Status**: mirrors `ContainerRegistry` — `LockedPresetID` XOR `LockedResources { DiskMB }`. No `LockedConfiguratorID`. Conditions identical (also without `NoConfiguratorAvailable`).

**Relationships / Lifecycle**: same pattern as `ContainerRegistry`.

---

### 1.6 `ContainerRegistryPreset` — DELETED

**API**: `containerregistry.timeweb.crossplane.io/v1alpha1`, kind `ContainerRegistryPreset`.

**Status**: removed. The CRD is dropped from `package/crds/`; the Go types are deleted from `apis/containerregistry/v1alpha1/preset_types.go`. Per FR-002 and the respec.

---

### 1.7 `Project` and `SshKey` — UNCHANGED apart from dual-PC reference

The MVP `Project` (`project.timeweb.crossplane.io`) and `SshKey` (`sshkey.m.timeweb.crossplane.io`) MRs keep their existing spec and status shapes. The only change is that their reconciler `Setup` registers both PC kinds via `managed.WithProviderConfigKinds(...)` so they accept either a same-namespace `ProviderConfig` or a cluster-scoped `ClusterProviderConfig`.

## 2. Internal (non-Kubernetes) entities

### 2.1 Resolver cache

**Location**: `internal/controller/shared/resolver`, process-local Go struct. Not a Kubernetes object. Not persisted.

**Purpose**: Bounded, TTL-keyed cache of upstream Timeweb catalog responses, plus a singleflight group for coalescing concurrent fetches.

**Shape** (sketch — concrete Go in the resolver-internal contract):

```go
type cacheKey struct {
    PCRef     PCRef     // {Kind, Name, Namespace}
    Dimension string    // dimension name, e.g. "ContainerRegistryPreset"
}

type cacheEntry struct {
    fetchedAt time.Time
    payload   any       // typed per dimension kind — see below
    err       error     // last fetch error (e.g. unauthorized; sticky until next successful fetch)
}

type Cache struct {
    mu       sync.RWMutex
    entries  map[cacheKey]*cacheEntry
    inflight singleflight.Group
    ttl      time.Duration   // configurable; default 5min
}
```

**Payload types by dimension kind**:

- **Preset**: `[]presetEntry { upstreamID int64; slug string; descriptionShort string; location string; rawFields map[string]any }`. Used to resolve `presetName` slugs to upstream IDs (FR-006).
- **Configurator**: `[]configuratorEntry { upstreamID int64; location string; diskType string; cpuFrequency string; isAllowedLocalNetwork bool; requirements requirementsBounds }`. Used to deterministically pick a configurator from operator-supplied `resources` fields (FR-007).
- **Enum**: `[]string`. Used to validate enum-dimension free-form string inputs (FR-017).

**Entry lifecycle**:

1. **Cold miss** → singleflight dispatches one upstream GET; result populates `payload`, sets `fetchedAt = now`.
2. **Warm hit (age < TTL)** → return cached payload.
3. **Warm expiry (age ≥ TTL)** → treat as cold miss on next access.
4. **Upstream 4xx on a previously-cached entry** (e.g. preset removed since last fetch) → invalidate entry; next access re-fetches (FR-013).
5. **Upstream 401/403** → store error in entry; sticky until the next successful fetch clears it. Callers see `CatalogUnauthorized` (FR-012).
6. **Upstream 5xx** → exponential backoff with retry; the calling reconcile sees a transient error and is requeued; cache entry unchanged (FR-012).
7. **Restart** → entire cache is cold; rebuilds lazily on demand (process-local by design).

**Concurrency**: all writes under `mu`; reads under `RWMutex.RLock`. Singleflight ensures a single in-flight upstream fetch per key (FR-011).

### 2.2 Dimension registry

**Location**: `internal/controller/shared/resolver/dimensions.go`. A small table mapping logical dimension names to:

- `kind` — DimensionPreset | DimensionConfigurator | DimensionEnum
- `fetcher` — closure that takes a `timeweb.Client` and returns the typed payload (one HTTP call)
- `responseFilter` — optional predicate applied to entries returned by `fetcher` (e.g. `type == "master"` for the K8s master preset dimension)
- `enumDerive` — for enum dimensions whose values are embedded in another endpoint's response, a derivation closure (e.g. `AvailabilityZone` derives from the K8s preset response)

**Initial registrations** (in scope for this feature):

| Dimension                       | Kind         | Endpoint (impl)                              | Filter / Derive    |
|---------------------------------|--------------|----------------------------------------------|--------------------|
| `ContainerRegistryPreset`       | Preset       | `GET /api/v1/container-registry/presets`     | —                  |
| `S3BucketPreset`                | Preset       | `GET /api/v1/presets/storages`               | —                  |

Per the **catalog-endpoint reality check** clarification (spec.md §Clarifications 2026-05-31), Container Registry and S3 Storage have **no configurator endpoint** upstream — both are platform-managed services with no operator-visible hypervisor/DC placement. The `resources` variant for those two MRs is therefore disk-only (`{ diskGB }` / `{ diskMB }`) and bypasses the configurator selection algorithm entirely. `ContainerRegistryConfigurator` and `S3BucketConfigurator` rows that existed in a prior draft of this table were removed because the corresponding `/api/v1/configurator/registries` and `/api/v1/configurator/storages` endpoints do not exist in `docs/openapi-timeweb.json`. The `DimensionConfigurator` kind itself remains in the resolver primitive for the forward-compat registrations below.

**Forward-compat registrations** (shipped now but not yet consumed; documents the K8s readiness commitment per SC-007):

| Dimension                       | Kind         | Endpoint (impl)                              | Filter / Derive             |
|---------------------------------|--------------|----------------------------------------------|-----------------------------|
| `ServerConfigurator`            | Configurator | `GET /api/v1/configurator/servers`           | —                           |
| `KubernetesMasterPreset`        | Preset       | `GET /api/v1/presets/k8s`                    | `entry.type == "master"`    |
| `KubernetesWorkerPreset`        | Preset       | `GET /api/v1/presets/k8s`                    | `entry.type == "worker"`    |
| `KubernetesVersion`             | Enum         | `GET /api/v1/k8s/k8s-versions`               | —                           |
| `KubernetesNetworkDriver`       | Enum         | `GET /api/v1/k8s/network-drivers`            | —                           |
| `AvailabilityZone`              | Enum         | (derived from K8s preset response)           | enumDerive over preset list |

Forward-compat registrations are wired into the registry but **only exercised by the K8s feature**. Their tests (registration smoke test, table-driven coverage) ship in this feature so the table doesn't drift before the K8s feature lands.

## 3. Relationships diagram

```
ClusterProviderConfig (cluster-scoped)        ProviderConfig (namespaced)
        │                                            │
        │ owns                                       │ owns
        ▼                                            ▼
ClusterProviderConfigUsage              ProviderConfigUsage
        │                                            │
        │  ────────────────  referenced by  ─────────│
        ▼              (explicit kind, no fallback)  ▼
                MR.spec.providerConfigRef
                ┌──────────────────────────┐
                │   ContainerRegistry      │  spec.forProvider: presetName XOR resources
                │   S3Bucket               │       → status.atProvider.{lockedPresetID | lockedResources}
                │   Project (unchanged)    │
                │   SshKey  (unchanged)    │
                └──────────────────────────┘
                          │
                          │ calls (during reconcile, on cache miss)
                          ▼
                  resolver.Resolve(...)
                          │
                          │ reads/writes
                          ▼
                  Cache (process-local)
                          │
                          │ on miss → singleflight → one GET
                          ▼
                  Timeweb HTTP API (catalog endpoints)
```

The cache is the only piece outside Kubernetes; every other entity lives as a Kubernetes CR.
