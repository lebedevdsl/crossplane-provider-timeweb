# Feature Specification: Internal Catalog Resolution & ProviderConfig Scoping

**Feature Branch**: `002-readonly-presets-design`

**Created**: 2026-05-20

**Last Respec**: 2026-05-31

**Status**: Draft

**Input**: User description: "respec for this feature, readonly resources should not be exposed as managed resources and instead those should be called by controller to validate user input only. preset_id configurator_id should be derived from stable fields, keep the design for Virtual Machine/Server (no configurator_id, location, other fields XOR preset name)"

## Clarifications

### Session 2026-05-31 (catalog-endpoint reality check)

- Q: Do Container Registry and S3 Bucket each have a configurator endpoint comparable to `/api/v1/configurator/servers`, so the `resources` variant can drive configurator selection the same way Server will? → A: **No.** Probing `docs/openapi-timeweb.json` (Phase 1 T004) confirms `/api/v1/configurator/servers` is the **only** configurator endpoint Timeweb exposes. Configurators model hypervisor / DC-level capability bounds — relevant for compute (Server, K8s control plane, K8s node group) but **not applicable to managed Container Registry or managed S3 Storage**, which are platform-managed services with no operator-visible hypervisor placement. Container Registry and S3 Bucket therefore have **one custom dimension only: disk size**. The `presetName XOR resources` shape is preserved — `resources` for these two MRs collapses to `{ diskGB }` (Container Registry) or `{ diskMB }` (S3 Bucket; upstream uses MB) plus the MR's existing non-sizing fields. No configurator resolution, no enum dimension checks, no `lockedConfiguratorID`. The full XOR + `lockedConfiguratorID` + `lockedResources` machinery is preserved in the spec for Server / KubernetesCluster / KubernetesNodeGroup, all of which DO have configurators. Resolver dimension registry initial entries (data-model.md §Dimension Registry) drop `ContainerRegistryConfigurator` and `S3BucketConfigurator`; the resolver's `Configurator` dimension kind itself stays in for the forthcoming Server / K8s feature.

### Session 2026-05-31 (terminology refactor)

- Q: Replace the term "axis" with what across the spec, plan, contracts, and quickstart? → A: **Use the literal field names `presetName` and `resources` instead of "axis".** The two sides of the XOR are called "sizing variants", referred to by their field name (e.g. "the `presetName` variant", "the `resources` variant"). The condition reason `AxisSwitchRequiresRecreate` is renamed to `SizingSwitchRequiresRecreate`. The CRD printer column previously called `AXIS` is renamed to `SIZING`, with values `presetName` or `resources`. Field names on the spec and status sides are unchanged (`presetName`, `resources`, `lockedPresetID`, `lockedConfiguratorID`, `lockedResources`).

### Session 2026-05-31 (K8s-readiness follow-up)

- Q: A future K8s Cluster MR has a control plane AND a list of worker node groups, each independently sized. How does the spec's `presetName XOR resources` shape extend to composite MRs? → A: **Every sizing block in `forProvider`, at any depth, uses the same `presetName XOR resources` shape.** The resolver runs independently per block. K8s Cluster becomes `forProvider: { controlPlane: <block>, nodeGroups: [<block>, …] }`. The spec rule (FR-004) generalizes from "the MR's `forProvider`" to "every sizing block on the MR." Composite MRs are a structural composition of the existing primitive, not a new shape.
- Q: K8s Cluster requires non-sizing readonly inputs — `k8sVersion`, `networkDriver`, `availabilityZone` — each a free-form string selecting one of a finite upstream set. How are these modelled? → A: **A third resolver kind: enum dimension.** Each enum dimension is `(upstream endpoint, returns string set)`. Validated at reconcile against the same `(providerConfigRef, dimension)` TTL cache used for presets/configurators. The operator types the value as a plain string field on `forProvider` (outside the XOR sizing block). Unknown values surface as `Synced=False, reason=<Dimension>NotFound` with a message listing valid values visible to that credential. K8s-specific dimensions in scope: `KubernetesVersion` (`/api/v1/k8s/k8s-versions`), `KubernetesNetworkDriver` (`/api/v1/k8s/network-drivers`), `AvailabilityZone` (derived from the appropriate Timeweb endpoint or, if no dedicated catalog endpoint exists, from a curated list inside the K8s preset response). No static CEL enums for these — runtime fetch keeps the provider aligned with Timeweb's catalog without a release.
- Q: How does the spec express the mapping from a logical dimension (e.g. K8s master preset) to its upstream URL, given Timeweb uses inconsistent path orders (`/api/v1/presets/k8s` vs `/api/v1/container-registry/presets`) and sometimes a single endpoint serves two logical dimensions via a response-side discriminator (K8s preset `type: master|worker`)? → A: **The dimension is a logical label; the spec commits the abstraction, not the URLs.** The resolver maps a dimension to `(upstream endpoint, optional response-side filter)` via a controller-internal table. Endpoint paths and discriminators are an implementation detail and do not appear normatively in this spec. New MR kinds extend the internal table without a spec edit.
- Q: How does the spec's sizing-locking rule (FR-009, FR-010) extend to composite MRs whose `forProvider` contains list-shaped sizing? → A: **Per-block locking.** Each sizing block locks its own variant (`presetName` or `resources`) independently. For list-shaped sizing, entries are identified by their operator-supplied `name` (immutable per entry). Adding a new list entry creates a new upstream child on the next reconcile; removing an entry deletes that upstream child via within-MR cascade; editing one entry's block triggers a within-variant PATCH or `Synced=False, reason=SizingSwitchRequiresRecreate` **scoped to that block only** — the rest of the MR is unaffected. Mixing variants across blocks on the same MR is explicitly permitted. *(Note after Q5: no in-scope or near-term MR actually uses list-shaped sizing — the K8s Cluster split-MR decision below moved node groups to their own MR kind. The per-block semantics remain as a forward-compatibility commitment so any future composite MR inherits a defined contract without a spec edit.)*
- Q: For the future K8s Cluster feature, does a Kubernetes cluster's worker node groups live inside a single composite `KubernetesCluster` MR (`nodeGroups[]` list), or are they separate `KubernetesNodeGroup` MRs each referencing their cluster? → A: **Separate MRs.** A `KubernetesCluster` MR carries control-plane sizing only (single sizing block). Each worker node group is its own `KubernetesNodeGroup` MR with a `clusterRef` pointing at a `KubernetesCluster` in scope, and its own single sizing block. This matches the Crossplane v2 one-MR-one-upstream-resource idiom already used by `ContainerRegistry → Project`, gives each node group independent reconcile/healing, and reuses the existing single-block XOR + sizing-lock semantics. Trade-off: an extra manifest per worker group; `KubernetesNodeGroup` reconciles wait on the referenced cluster reaching `Ready=True` before their first successful create (same wait-on-prerequisite pattern as today's MRs on `Project`).

### Respec — Session 2026-05-31

This session **supersedes** the prior catalog-as-CRDs design. The earlier design exposed read-only Timeweb catalog data as observable CRDs (`Catalog`, `ContainerRegistryPreset`, `S3BucketPreset`, `ServerPreset`, and matching `*Configurator` siblings). That entire surface is dropped. Catalog data becomes an internal concern of the managed-resource controllers: they fetch and cache it just-in-time to validate operator input and resolve upstream IDs.

- Q: Do readonly Timeweb catalog resources (presets, configurators) surface as Kubernetes objects? → A: **No.** No `Catalog`, no `*Preset`, no `*Configurator` CRDs. The controller fetches catalog data on demand from the Timeweb API and uses it only to (a) validate operator-supplied sizing inputs at reconcile time, and (b) resolve a stable user-facing identifier (`presetName` or a sizing block) to the upstream `preset_id` / `configurator_id` required by the Timeweb create/update endpoints.
- Q: How does the operator express sizing on a managed resource? → A: **One of two mutually exclusive, stable shapes**, enforced by CEL `oneOf` on `forProvider`. Either `presetName: <string-slug>` OR an embedded `resources: { … }` block carrying human-meaningful fields (location, diskType, cpu, ramMB, diskGB, gpu, bandwidthMbps, cpuFrequencyTier, enableLocalNetwork, storageClass, etc., per MR kind). The operator never types `preset_id` or `configurator_id`; those are derived by the controller from the stable fields and recorded in `status.atProvider` as `lockedPresetID` / `lockedConfiguratorID` after the first successful create.
- Q: Is this XOR shape the same for the future Server (Virtual Machine) MR? → A: **Yes — same pattern.** Server's spec is `presetName XOR resources{location, diskType, cpu, ramMB, diskGB, gpu, bandwidthMbps, cpuFrequencyTier, enableLocalNetwork, …}`. No `configurator_id`, no `location` field at the top level — `location` lives inside `resources`. The controller picks the right configurator deterministically from the stable fields. ContainerRegistry and S3Bucket use the same shape with a reduced field set appropriate to their dimension.
- Q: How does the controller avoid hammering the Timeweb catalog endpoints on every reconcile? → A: **In-controller TTL cache per ProviderConfig**, default 5 minutes (configurable via controller flag, minimum 1 minute, maximum 1 hour). Cache key is `(providerConfigRef, dimension)`. A cache miss triggers a single upstream GET; concurrent reconciles for the same key coalesce onto one in-flight request. Cache is purely process-local; restart re-warms it lazily.
- Q: What replaces the existing `ContainerRegistryPreset` CRD and its hand-rolled `preset_reconciler.go` timer goroutine? → A: **Both are deleted outright.** No replacement controller, no migrated group. The MVP CRD `ContainerRegistryPreset` is removed from `package/`; the `manager.Runnable` poller is removed from `internal/controller/containerregistry/`. Any in-cluster CRs from the MVP are GC'd when the CRD is uninstalled; since they were observe-only and re-derivable, no data is lost.
- Q: How does an operator discover the available `presetName` values without `kubectl get`? → A: **Provider documentation and the Timeweb dashboard.** A short `docs/presets.md` lists the in-cluster slug rule (`<short>-<location>` for presets) and points operators at the Timeweb dashboard for the live catalog. The controller's reconcile-time error messages on `PresetNotFound` echo back the list of valid slugs visible to the MR's `providerConfigRef` (capped to e.g. 20 with a link/hint), giving an inline, actionable signal without requiring a discovery CRD.
- Q: What's the slug rule the operator types into `presetName`? → A: **`<short>-<location>`** (lowercase, non-`[a-z0-9-]` runs collapsed to `-`), matching the upstream catalog entry's `description_short` and `location`. Examples: `start-ru-1`, `pro-pl-1`. Collision disambiguator: if two upstream entries within the dimension collide on this slug at resolution time, the controller surfaces `Synced=False, reason=PresetAmbiguous` with both upstream IDs listed in the message; the operator must use the explicit suffixed form `<short>-<location>-<id>` (e.g. `start-ru-1-199`) to disambiguate. Ambiguity is expected to be rare.
- Q: What does the operator-facing failure surface look like when stable fields don't match any upstream configurator? → A: **`Synced=False, reason=NoConfiguratorAvailable`** with a message naming the requested `(location, diskType, cpu, ramMB, diskGB, gpu, …)` and at least one candidate configurator that was filtered out, including which bound(s) excluded it. Operator can fix the manifest without external docs.
- Q: What happens on a sizing-variant switch (`presetName` ↔ `resources`) post-create? → A: *(unchanged from prior session)* `Synced=False, reason=SizingSwitchRequiresRecreate` — Timeweb rejects switching between the two upstream sizing modes on an existing resource; the controller surfaces it; the operator must delete and recreate. Within-variant changes (different `presetName` slug, different `resources.cpu`) PATCH normally.
- Q: Is admission-time CEL validation of `resources` against live catalog data in scope? → A: **No.** Catalog data lives outside Kubernetes; CEL has no read-side hook to it. The controller validates at reconcile time and surfaces typed conditions. Static CEL rules (range/enum on individual fields) are in scope where they don't require live catalog data.
- Q: Does `ProviderConfig` stay cluster-scoped or become namespaced? → A: **Both — namespaced `ProviderConfig` and cluster-scoped `ClusterProviderConfig`.** Same decision as the prior session. The MVP's cluster-scoped `ProviderConfig` is renamed to `ClusterProviderConfig`; a new namespaced `ProviderConfig` is added; existing controllers (`Project`, `SshKey`) accept either via the Crossplane v2 dual-reference convention. This is groundwork independent of catalog resolution but bundled here because the MR refactor in US1 already touches every controller and the v2 namespaced model is the long-term shape.
- Q: Are downstream artifacts (`plan.md`, `tasks.md`, `research.md`, `data-model.md`, `contracts/`) under this feature directory still valid? → A: **No — they describe the dropped catalog-CRD design.** Re-run `/speckit-plan` and `/speckit-tasks` after this respec lands to regenerate them. The `001-mvp-scaffolding/` artifacts (Project/SshKey kinds, error classification, external-name conventions) remain authoritative and carry forward unchanged.

## User Scenarios & Testing *(mandatory)*

### User Story 1 — Size and create managed resources with stable, human-meaningful fields (Priority: P1)

A platform operator declares a `ContainerRegistry` or `S3Bucket` (or, post-feature, a `Server`) by writing the sizing they actually understand: location, disk size, optionally CPU/RAM and storage class. They never see, type, or copy a `preset_id` or `configurator_id`. The controller fetches the relevant Timeweb catalog data on demand, validates the operator's fields against live upstream bounds, picks the right configurator deterministically, and creates the resource. The chosen ID is recorded in `status.atProvider` so future reconciles consult it and refuse silent switches between the `presetName` and `resources` sizing variants.

This story also covers the bundled prerequisite refactor: the MVP's cluster-scoped `ProviderConfig` is renamed to `ClusterProviderConfig`, a new namespaced `ProviderConfig` is introduced, and existing controllers (`Project`, `SshKey`) accept either via the standard Crossplane v2 dual-reference convention.

**Why this priority**: This is the entire user-facing payoff. Operators get meaningful sizing on the two MRs that exist in code today (`ContainerRegistry`, `S3Bucket`) with no catalog mental model and no in-cluster catalog infrastructure to install. The ProviderConfig dual-scope refactor lands here because every MR touched in this story already needs an updated `providerConfigRef` handler.

**Independent Test**: With the provider installed and authenticated, an operator applies a namespaced `ProviderConfig`, then applies a `ContainerRegistry` manifest carrying only `forProvider.resources: { location: ru-1, diskGB: 5 }` and an `S3Bucket` manifest carrying `forProvider.resources: { location: ru-1, diskGB: 30, storageClass: hot }`. Both resources appear upstream within two minutes with `Ready=True` and `Synced=True`. The operator patches `diskGB` upward and the upstream resource is updated in place. The operator attempts to switch the spec from `resources` to `presetName` and sees `Synced=False, reason=SizingSwitchRequiresRecreate`. No new CRDs beyond the refactored MR CRDs and the dual-scope ProviderConfig pair exist in the cluster throughout the test.

**Acceptance Scenarios**:

1. **Given** a namespaced `ProviderConfig` referencing a valid token Secret exists in namespace `team-a`, **When** the operator applies a `ContainerRegistry` in `team-a` with `forProvider.resources: { location: ru-1, diskGB: 5 }`, **Then** the controller resolves a configurator from a single upstream GET (or warm cache), records `lockedConfiguratorID` in `status.atProvider`, creates the registry upstream, and the resource reports `Ready=True, Synced=True` within two minutes.
2. **Given** the registry from (1) is `Ready=True`, **When** the operator patches `forProvider.resources.diskGB` from 5 to 10, **Then** the controller submits a within-variant PATCH to Timeweb and the resource returns to `Ready=True` without recreating.
3. **Given** the registry from (1) is `Ready=True`, **When** the operator patches the spec to replace `resources` with `presetName: anything`, **Then** the resource transitions to `Synced=False` with `reason=SizingSwitchRequiresRecreate` and the upstream resource is unchanged; the operator must `kubectl delete` and reapply to switch sizing variants.
4. **Given** the operator's `forProvider.resources.cpu` requests a value outside every available configurator's `requirements.cpu_max` for the requested `(location, diskType)`, **When** the controller attempts resolution, **Then** the resource transitions to `Synced=False` with `reason=NoConfiguratorAvailable` and a message naming the requested values plus the bound that ruled out at least one candidate configurator.
5. **Given** the existing cluster-scoped `ProviderConfig` from the MVP exists at upgrade time, **When** the operator installs this feature's CRDs, **Then** the existing object is renamed/migrated to `ClusterProviderConfig` (or coexists during a transition window) and the `Project` and `SshKey` MRs already in the cluster continue to reconcile against it without manual intervention.
6. **Given** ten `ContainerRegistry` MRs in the same namespace under the same `ProviderConfig` reconcile within a minute, **When** the cache is empty, **Then** the controller issues at most one upstream catalog GET per dimension (configurator list) — concurrent reconciles for the same `(providerConfigRef, dimension)` coalesce.

### User Story 2 — Pick a named tariff by string slug (Priority: P2)

A platform operator wants to lock to a specific Timeweb tariff ("Start" tier, "Standard" tier) by its named slug rather than asserting that their raw `resources` will resolve to the same one. They write `forProvider.presetName: start-ru-1` directly. The controller fetches the preset list for the MR's `providerConfigRef`, resolves `start-ru-1` to its upstream `preset_id`, locks the ID into `status.atProvider`, and creates the resource using the upstream `preset_id` path (not the `configuration{configurator_id, …}` path). If the operator typed a slug that doesn't exist, the controller surfaces a `PresetNotFound` condition whose message lists the valid slugs (capped) so the next edit is informed.

**Why this priority**: Named-tier semantics matter to operators who care about Timeweb's published tariffs ("the Start tariff, whatever Timeweb sets it to") rather than reverse-engineering equivalent `resources` blocks. It ships on top of US1 — operators who don't want named tariffs ignore `presetName` and use `resources`. There is no separate discovery CRD; discovery happens via `docs/presets.md` + dashboard + the `PresetNotFound` error message.

**Independent Test**: With US1 working, the operator applies a `ContainerRegistry` carrying `forProvider.presetName: start-ru-1`. The controller resolves the slug from a fresh or cached upstream catalog response, records `lockedPresetID` in `status.atProvider`, and the resource reaches `Ready=True` within two minutes. The operator applies a second MR with `presetName: nonexistent-slug` and sees `Synced=False, reason=PresetNotFound` with a message listing currently-valid slugs. The operator corrects the slug and the MR reaches `Ready=True` on the next reconcile. No CRDs beyond the refactored MR CRDs and the ProviderConfig pair exist.

**Acceptance Scenarios**:

1. **Given** the upstream catalog contains a preset with `description_short=Start` and `location=ru-1`, **When** the operator applies an MR with `forProvider.presetName: start-ru-1`, **Then** the controller resolves to that upstream `preset_id`, records `lockedPresetID` in `status.atProvider`, calls Timeweb using the `preset_id` upstream path, and the resource reaches `Ready=True` within two minutes.
2. **Given** a consumer MR is `Ready=True` with `lockedPresetID` set, **When** Timeweb's catalog renames or re-IDs that preset such that the same slug now maps to a new upstream ID, **Then** the controller's next reconcile detects the change, PATCHes upstream within the `presetName` sizing variant, and the MR remains `Ready=True`.
3. **Given** the operator's MR carries `presetName` that does not exist in the upstream catalog for the MR's `providerConfigRef`, **When** the controller attempts resolution, **Then** the MR transitions to `Synced=False, reason=PresetNotFound` with a message naming the unresolved slug and listing up to 20 currently-valid slugs visible to the MR's `providerConfigRef`.
4. **Given** two upstream preset entries collide on the slug `start-ru-1` (rare but possible), **When** the controller attempts resolution, **Then** it surfaces `Synced=False, reason=PresetAmbiguous` with both upstream IDs in the message; the operator can disambiguate by using the explicit `start-ru-1-<id>` form.
5. **Given** the operator's MR uses `presetName` AND `resources` simultaneously, **When** they `kubectl apply`, **Then** the apply is rejected at admission by the CEL `oneOf` rule with a clear schema-level error.

### User Story 3 — Multi-tenant credential isolation across ProviderConfigs (Priority: P3)

A multi-team cluster runs two `ProviderConfig`s (in different namespaces, or both namespaced and one cluster-scoped) tied to distinct Timeweb accounts or differently-scoped tokens against the same account. Each MR resolves its sizing against the catalog data visible to its own `providerConfigRef`. The in-controller cache is keyed by `(providerConfigRef, dimension)` so credential A's view never leaks into credential B's resolution decisions. If one credential's token is scoped to deny a catalog endpoint, MRs under that credential surface a specific `CatalogUnauthorized` condition without affecting MRs under the other credential.

**Why this priority**: Today's MVP `PresetReconciler.PCName: string` hardcodes a single ProviderConfig and silently misbehaves with multiple. Removing that reconciler (per the respec) eliminates the field, but the requirement to isolate per-PC remains — it's now a property of the cache key and the per-reconcile resolver. US3 makes the design honest about multi-credential clusters.

**Independent Test**: With US1 + US2 working, the operator applies a namespaced `ProviderConfig` `pc-a` in namespace `team-a` and a `ClusterProviderConfig` `pc-b`. They apply MRs under each and observe each resolves independently. They construct a token scoped to deny `/api/v1/configurator/registries` (simulated by a 403 intercept in the test harness) and assign it to `pc-b`. MRs under `pc-a` continue to reconcile normally; MRs under `pc-b` that need configurators surface `Synced=False, reason=CatalogUnauthorized` with a message naming the denied endpoint and pointing at the token scope.

**Acceptance Scenarios**:

1. **Given** a namespaced `ProviderConfig` `pc-a` in `team-a` and a `ClusterProviderConfig` `pc-b`, **When** an MR in `team-a` references `pc-a` and another MR in `team-b` references `pc-b`, **Then** each MR resolves against catalog data fetched under its own credentials, with no cross-PC cache reads.
2. **Given** an MR references `pc-b` and `pc-b`'s token is scoped to deny one of the catalog endpoints, **When** the controller attempts resolution, **Then** the MR transitions to `Synced=False, reason=CatalogUnauthorized` with a message identifying the endpoint and credential reference; MRs under `pc-a` remain unaffected.
3. **Given** a `ProviderConfig` is deleted while live MRs reference it, **When** Kubernetes garbage collection runs, **Then** the existing MR finalizer behavior (carried forward from the MVP) gates deletion as already defined; this spec adds no new finalizer logic.
4. **Given** the cluster-scoped `ClusterProviderConfig` and the namespaced `ProviderConfig` coexist with overlapping names, **When** an MR's `providerConfigRef` resolves, **Then** the Crossplane v2 dual-reference rule (namespaced-first within the MR's namespace, falling back to cluster-scoped) applies — no new resolution semantics are introduced.

### Edge Cases

- **Upstream catalog endpoint returns a transient 5xx**: controller retries with exponential backoff (capped); MR stays in its prior status (`Synced=Unknown` if first reconcile, or unchanged if it had previously synced). No MR transitions to terminal failure on a single upstream blip.
- **Cache hit serves a slug that was removed upstream after the last fetch**: controller detects the mismatch on the upstream create/PATCH call (Timeweb returns 4xx for the stale preset_id), invalidates the cache entry, and refetches on the next reconcile. The MR surfaces `Synced=False, reason=PresetNotFound` only after the refetch confirms removal — not from a stale cache.
- **Operator's MR `resources` block matches a fixed-tariff preset by coincidence**: still goes through the configurator path. The XOR is by operator-declared sizing variant, not by best-fit substitution.
- **Operator changes within `resources` such that no configurator now matches** (e.g. raises `cpu` above all bounds): controller leaves the locked configurator on the upstream resource and surfaces `Synced=False, reason=NoConfiguratorAvailable` — the prior resource keeps running upstream; only the change is rejected. The operator either rolls back the spec or deletes and recreates.
- **MVP-era `ContainerRegistryPreset` CRs exist in the cluster at upgrade**: removing the CRD via the new package GCs them; no operator action required (they are observe-only and carry no state worth preserving).
- **Two `ProviderConfig`s reference the same Timeweb account**: catalog responses for them are identical but cached independently (different cache keys). No deduplication; correctness wins over the marginal request savings.
- **Cluster restarts the controller mid-reconcile**: cache is process-local and cold on restart; first reconciles after restart pay one upstream GET per `(providerConfigRef, dimension)` and warm the cache. No persistent state to migrate.
- **CRD schema migration for the refactored MRs**: the MVP MR shape (raw `preset_id` / `configuration` fields) is replaced. Since the MVP code has no external consumers (per assumptions), operators reapply manifests in the new shape; no conversion webhook is needed.

## Requirements *(mandatory)*

### Functional Requirements

- **FR-001**: The provider MUST publish a namespaced `ProviderConfig` CRD and a cluster-scoped `ClusterProviderConfig` CRD under the existing `timeweb.crossplane.io` group. The MVP's current cluster-scoped `ProviderConfig` MUST be renamed to `ClusterProviderConfig`. All existing managed resources (`Project`, `SshKey`, and the MRs refactored in this spec) MUST accept either via Crossplane v2's dual-reference convention on `spec.providerConfigRef`.
- **FR-002**: The provider MUST NOT publish any Kubernetes CRDs representing read-only Timeweb catalog data. No `Catalog`, no `*Preset`, no `*Configurator` kinds. The MVP-era `ContainerRegistryPreset` CRD MUST be removed from the provider package.
- **FR-003**: The MVP-era `internal/controller/containerregistry/preset_reconciler.go` `manager.Runnable` timer loop and any `PCName` global state MUST be deleted; no replacement controller is added for catalog data.
- **FR-004**: Every sizing block on a consuming managed resource (`ContainerRegistry`, `S3Bucket`, and the future `Server`, `Kubernetes Cluster`) MUST expose exactly one of `presetName: <string>` or `resources: { … }`, enforced by a CEL `oneOf` validation rule on the CRD. The rule applies at any depth: single-block MRs (e.g. `ContainerRegistry`) carry one sizing block at `spec.forProvider`; composite MRs (e.g. `Kubernetes Cluster`) carry multiple sizing blocks (e.g. `spec.forProvider.controlPlane`, `spec.forProvider.nodeGroups[*]`), each governed by the same XOR rule. The controller's resolver runs independently per block. Neither side MUST expose `preset_id` or `configurator_id` to the operator; both are derived by the controller per block and recorded in the corresponding `status.atProvider` block only.
- **FR-005**: For `ContainerRegistry` and `S3Bucket`, `resources` MUST carry the stable user-facing fields appropriate to the dimension (at minimum: `location`, `diskGB`; for `S3Bucket` also `storageClass`). For the future `Server`, `resources` MUST carry `location`, `diskType`, `cpu`, `ramMB`, `diskGB`, optional `gpu`, `bandwidthMbps`, `cpuFrequencyTier`, `enableLocalNetwork`. No `configurator_id` or `location-outside-resources` field on `Server`.
- **FR-006**: When a sizing block's `presetName` is set, the controller MUST resolve the string slug to an upstream `preset_id` by fetching the relevant preset dimension's upstream endpoint under the MR's `providerConfigRef` credentials, using an in-controller TTL cache (default 5 minutes, configurable 1 min – 1 hour) keyed by `(providerConfigRef, dimension)`. The mapping from a logical dimension to its concrete upstream URL (and any response-side filter such as a `type` discriminator) is a controller-internal detail and is not normative in this spec. Unresolved slugs MUST surface as `Synced=False, reason=PresetNotFound` with a message naming the unresolved slug and listing up to 20 currently-valid slugs visible to that credential. Ambiguous slugs (rare collisions) MUST surface as `Synced=False, reason=PresetAmbiguous` with the colliding upstream IDs in the message.
- **FR-007**: When `forProvider.resources` is set, the controller MUST deterministically pick a configurator from the upstream catalog data (fetched and cached per FR-006's caching rules). Selection algorithm: (1) hard-filter by `(location, diskType, enableLocalNetwork, cpuFrequencyTier?)`; (2) capability-filter by `requirements.gpu_max ≥ gpu` and bounds containing the requested `(cpu, ramMB, diskGB, bandwidthMbps)`; (3) tightest fit by ascending `(max_cpu, max_ramMB, max_diskGB)`; (4) tiebreaker by lowest `configurator_id`. Zero matches MUST surface as `Synced=False, reason=NoConfiguratorAvailable` with a message naming the requested values and the bound that ruled out at least one candidate configurator.
- **FR-008**: The slug rule the operator types into `presetName` is `<short>-<location>` (lowercase, non-`[a-z0-9-]` runs collapsed to `-`), matched against the upstream entry's `description_short` and `location`. The explicit suffixed form `<short>-<location>-<upstream-id>` MUST also resolve correctly (for disambiguation on collision).
- **FR-009**: On first successful create of a sizing block's upstream resource, the controller MUST record the resolved upstream identifier in that block's `status.atProvider` slot (`lockedPresetID` when the operator used the `presetName` variant, or `lockedConfiguratorID` plus the locked `cpu/ramMB/diskGB` when they used the `resources` variant). Subsequent reconciles MUST consult the per-block locked value and refuse to switch that block's sizing variant. Within-variant changes (different preset slug, or different `cpu`/`ramMB`/`diskGB` under the same configurator) MUST be PATCHed normally **scoped to the changed block**. For list-shaped sizing (e.g. `nodeGroups[*]`), list entries MUST be identified by their operator-supplied `name` (which MUST be immutable per entry once locked). Adding a new entry creates a new upstream child; removing an entry deletes the corresponding upstream child via within-MR cascade.
- **FR-010**: An attempted sizing-variant switch (`presetName` ↔ `resources`) on an existing sizing block MUST surface as `Synced=False, reason=SizingSwitchRequiresRecreate` with a message instructing the operator to delete and recreate the *affected block* (which, for composite MRs, may mean recreating only one list entry rather than the entire MR — operationally the operator removes the entry from the list, applies, then re-adds it with the new sizing variant). The upstream resource for the affected block MUST NOT be modified. Other blocks on the same MR MUST continue to reconcile normally.
- **FR-011**: Concurrent reconciles for the same `(providerConfigRef, dimension)` cache key MUST coalesce onto a single in-flight upstream GET. A controller restart cold-starts the cache and re-warms it lazily on subsequent reconciles.
- **FR-012**: Catalog endpoints returning 401/403 MUST surface as `Synced=False, reason=CatalogUnauthorized` with a message naming the denied endpoint and the credential reference. Catalog endpoints returning 5xx MUST be retried with exponential backoff; the MR MUST NOT terminally fail on a single transient upstream error.
- **FR-013**: A stale cache entry that produces an upstream 4xx (e.g. preset removed since the last fetch) MUST cause the controller to invalidate that cache entry and refetch on the next reconcile. The MR MUST NOT surface `PresetNotFound` from a stale cache alone; the refetch must confirm the removal.
- **FR-014**: The provider MUST ship a short operator-facing reference at `docs/presets.md` documenting (a) the slug rule, (b) how to find available presets in the Timeweb dashboard, (c) the `presetName` vs `resources` XOR and the locked-sizing behavior. No discovery CRD is required.
- **FR-015**: The cluster-scoped `ClusterProviderConfig` and namespaced `ProviderConfig` MUST be designed such that no schema changes are required to introduce additional namespaced or cluster-scoped MR kinds in future features.
- **FR-016**: The internal catalog cache and resolver MUST support three dimension kinds: (a) **preset** — returns a list of preset entries, resolves a `presetName` string to an upstream `preset_id`; (b) **configurator** — returns a list of configurator entries with `requirements` bounds, picks one deterministically per FR-007; (c) **enum** — returns a finite set of valid strings (e.g. K8s versions, CNI drivers, availability zones), validates that an operator-supplied free-form string is a member. All three kinds share the `(providerConfigRef, dimension)` cache key, the TTL semantics (FR-011), the unauthorized handling (FR-012), and the stale-cache invalidation rule (FR-013). Each MR kind declares which dimensions it consumes; the resolver is dimension-agnostic.
- **FR-017**: An MR kind MAY carry plain-string fields on `forProvider` that are validated against an enum dimension at reconcile time (outside the `presetName XOR resources` sizing block). Unresolved values MUST surface as `Synced=False, reason=<Dimension>NotFound` with a message naming the unresolved value and listing up to 20 currently-valid values visible to the MR's `providerConfigRef`. This is the contract the future Kubernetes Cluster MR will use for `k8sVersion`, `networkDriver`, and `availabilityZone`.

### Key Entities

- **ProviderConfig** (namespaced, group `timeweb.crossplane.io`): Crossplane v2 namespaced credential reference; carries `spec.credentials.secretRef`. Referenced by namespaced MRs in the same namespace.
- **ClusterProviderConfig** (cluster-scoped, group `timeweb.crossplane.io`): cluster-scoped sibling, renamed from today's MVP `ProviderConfig`. Referenced by cluster-scoped MRs (and by namespaced MRs via the dual-reference fallback).
- **Refactored consuming MRs** (`ContainerRegistry`, `S3Bucket`, future `Server`, future `KubernetesCluster`, future `KubernetesNodeGroup`): each carries one or more sizing blocks. In-scope kinds and near-term planned kinds all use **single-block** sizing at `spec.forProvider`. The spec's per-block rule (FR-004, FR-009, FR-010) is forward-compatible with composite MRs whose `forProvider` contains list-shaped sizing, but no such MR exists in the current roadmap. Every sizing block uses `presetName: <string-slug>` XOR `resources: { location, diskGB, … kind-specific stable fields }`. `status.atProvider` mirrors the spec shape, carrying `lockedPresetID` or `lockedConfiguratorID` plus locked sizing per block. MRs MAY additionally carry enum-validated string fields outside the XOR sizing blocks (e.g. on the future `KubernetesCluster`: `k8sVersion`, `networkDriver`, `availabilityZone`). The future `KubernetesNodeGroup` MR carries a `clusterRef` to its parent `KubernetesCluster` and reconciles only once the referenced cluster reports `Ready=True`.
- **Internal catalog cache** (not a Kubernetes object): in-controller per-process structure keyed by `(providerConfigRef, dimension)` holding the last upstream response and its fetch timestamp, where a dimension is one of **preset** | **configurator** | **enum**. Used for `presetName` resolution, configurator selection, and enum-field validation. Configurable TTL, coalesces in-flight fetches.

## Success Criteria *(mandatory)*

### Measurable Outcomes

- **SC-001**: An operator who has applied a namespaced `ProviderConfig` can create a working `ContainerRegistry` and a working `S3Bucket` using only `forProvider.resources` (stable fields, no IDs) in under five minutes from `kubectl apply` to `Ready=True`, with no catalog-related CRDs in the cluster.
- **SC-002**: An operator who applies an MR with `forProvider.presetName: <valid-slug>` sees `Ready=True` within two minutes; an operator who types an unrecognized slug sees a `Synced=False, reason=PresetNotFound` condition with a message that lists valid slugs and lets them correct the manifest without consulting external documentation.
- **SC-003**: When the operator submits an MR with `resources` that no available configurator can satisfy, the controller surfaces `Synced=False, reason=NoConfiguratorAvailable` within two reconciliation cycles, naming both the operator's requested values and the bound of at least one candidate configurator that was ruled out.
- **SC-004**: Under a load of 50 MRs reconciling within one minute against a single `ProviderConfig`, the controller issues at most one upstream catalog GET per dimension per TTL window (verified by a request-counter integration test); concurrent reconciles for the same cache key coalesce.
- **SC-005**: A cluster running two `ProviderConfig`s against different Timeweb accounts produces independent resolution decisions for MRs under each — verified by an integration test that gives `pc-a` a permissive token and `pc-b` a scoped token and asserts the MRs under each surface the appropriate per-credential conditions.
- **SC-006**: A future `Server` MR can be added with no spec changes to the dual-scope `ProviderConfig` pair and no spec changes to the internal catalog cache — only a new MR CRD whose `spec.forProvider` reuses the same `presetName XOR resources{ … }` shape established here. Verified by reviewing the diff of the follow-up Server feature against the artifacts produced by this spec.
- **SC-007**: The future `KubernetesCluster` + `KubernetesNodeGroup` MR pair can be added with no spec changes to the dual-scope `ProviderConfig` pair, the internal catalog cache, or the resolver primitive. Specifically: (a) `KubernetesCluster.spec.forProvider` reuses the single-block `presetName XOR resources` shape for the control plane; (b) `KubernetesCluster` declares enum dimensions `k8sVersion`, `networkDriver`, `availabilityZone` against this spec's enum-dimension resolver kind; (c) `KubernetesNodeGroup.spec.forProvider` reuses the single-block XOR for its worker sizing and carries a `clusterRef` to its parent `KubernetesCluster`. Verified by walking the K8s Cluster create-body and node-group create-body fields (per `docs/openapi-timeweb.json`'s `POST /api/v1/k8s/clusters` and `POST /api/v1/k8s/clusters/{cluster_id}/groups`) against this spec's three resolver dimension kinds + the standard cross-MR reference pattern, and confirming every required field maps to one of them.

## Assumptions

- The Timeweb API behaves as described in the vendored `docs/openapi-timeweb.json` for the catalog endpoints (`/api/v1/presets/{servers,storages,balancers,…}`, `/api/v1/configurator/{servers,…}`, `/api/v1/container-registry/presets`). Endpoint paths discovered to differ from the schema during implementation are resolved by curl-probing the live API; the public contract (this spec) remains the source of truth for operator-facing shape.
- Whether Timeweb's API strictly enforces `requirements.{min,step,max}` server-side is not confirmed. The provider validates against these bounds regardless — clear reconcile-time errors are preferable to opaque 400s from upstream.
- The Crossplane v2 dual-reference convention (a namespaced MR's `spec.providerConfigRef` can resolve to either a same-namespace `ProviderConfig` or a cluster-wide `ClusterProviderConfig`) is honoured by `crossplane-runtime` at the version this provider tracks; if not, FR-001 escalates to a "wait for runtime support" risk.
- Admission-time CEL validation of `forProvider.resources` against live catalog data is **out of scope** — CEL has no live read-side into the Timeweb API. Static CEL rules (per-field type/enum/range) are in scope where they don't require live catalog data.
- The MVP code shipped on `main` has no external consumers — removing the `ContainerRegistryPreset` CRD and the `preset_reconciler.go` `Runnable`, and replacing the MVP MR shape, do not require backwards-compatibility shims.
- The xpkg lint allow-list (CRD/MRD/webhook configs only in `package/`) continues to apply; this feature does not require any new admission webhook in `package/` (CEL on CRDs is sufficient for the `oneOf` rule).
- The Timeweb error classification, immutable-field inventory, and external-name conventions established in `specs/001-mvp-scaffolding/research.md` (R-2, R-3, R-4) carry forward unchanged for the refactored MRs.
- Downstream artifacts in this feature directory (`plan.md`, `tasks.md`, `research.md`, `data-model.md`, `contracts/`, `quickstart.md`) were generated against the prior catalog-CRD design and are stale after this respec. They MUST be regenerated by re-running `/speckit-plan` and `/speckit-tasks` before implementation continues.
