# Feature Specification: Cloud Server + Private Network MRs

**Feature Branch**: `003-server-mr-and-network`

**Created**: 2026-06-01

**Status**: Draft

**Input**: User description: "Create Server Managed resource and all the dependencies including network and connect those via the network ref, use timeweb API, terraform provider timeweb as examples and implement required Managed resources to run the server. Simplify first not all details are required."

## Clarifications

### Session 2026-06-01 (initial /speckit-specify pass)

- Q: Which Timeweb server family is in scope? → A: **Cloud servers only** (`/api/v1/servers` + `/api/v1/configurator/servers` + `/api/v1/presets/servers` + `/api/v1/os/servers`). **Dedicated servers** (`/api/v1/dedicated-servers`) are explicitly out of scope for this feature — the dashboard's "Create Server" flow this feature mirrors is the cloud-server flow, not the dedicated-server flow.
- Q: Which sizing path does the first iteration support — the dashboard's "Фиксированная" (Fixed preset) tab or "Произвольная" (Custom configurator) tab? → A: **Fixed preset only** (`presetName` resolving to upstream `preset_id`). The custom-configurator path (`resources{cpu, ramMB, diskGB, …}` resolving to `configurator_id`) is registered as a forward-compat dimension (already done in feature 002) but not exposed on the v0.1 `Server` CRD. Justified by the user's "simplify first not all details are required" directive; matches the most-common dashboard flow.
- Q: How do operators reference an existing private network from a Server MR? → A: **Crossplane-style cross-resource reference** (`forProvider.networkRef: {name: <network-mr-name>}` and `forProvider.networkSelector: {matchLabels: …}`) plus a flat `forProvider.networkID: <vpc-id>` escape hatch for importing a VPC not managed by the provider. The reference resolves to the upstream VPC ID via the standard crossplane-runtime reference resolver. Same pattern as the future `KubernetesNodeGroup` → `KubernetesCluster` ref already specified in feature 002.

### Session 2026-06-01 (network group commitment)

- Q: Should this feature expand v0.3 scope to add any additional network-class MRs (Balancer, FirewallRule/SecurityGroup), or just record the API-group home for the future and ship the simpler scope first? → A: **Forward-compat commitment only.** `network.m.timeweb.crossplane.io/v1alpha1` is locked as the canonical API group for the entire network-resource family: `Network` (VPC, v0.3), `FloatingIP` (v0.3), `Router` (future), `Balancer` (future — dashboard image #3 "Создать балансировщик"), `FirewallRule` / `SecurityGroup` (future). No additional MRs ship in v0.3; the commitment is documented so future features extend the existing group instead of fragmenting into `loadbalancer.m.…` / `firewall.m.…` / etc. The `compute.m.timeweb.crossplane.io` group remains for VM-class kinds only (`Server` today; `Disk`, `Backup`, `Snapshot`, etc. in future features). See new Assumption added below.

### Session 2026-06-01 (FloatingIP reference reversal + custom-server deferral)

Mid-implementation review (after Phase 1+2 of /speckit-implement landed) surfaced two course corrections:

- Q: Which direction should the FloatingIP↔Server reference run — IP-owns-binding (FloatingIP.serverRef authoritative, original FR-016/FR-017) or Server-consumes-IP (Server.floatingIPRefs authoritative, FloatingIP becomes pure allocation)? → A: **Reverse to Server-consumes-IP.** `Server.forProvider.{floatingIPRefs, floatingIPSelector, floatingIPIDs}` becomes the authoritative binding trio (CEL-enforced at-most-one). `FloatingIP.forProvider.{serverRef, serverSelector, serverID}` is **removed entirely** — FloatingIP becomes pure allocation (location + isDDoSGuard + optional comment + optional availabilityZone). The Server controller owns `POST /floating-ips/{id}/bind` + `unbind` calls; FloatingIP controller owns only allocation/release. Justified by: (a) every major cloud provider follows this direction in their Crossplane providers (GCP `compute.Address` referenced from `compute.Instance.network_interface[].nat_ip`; Azure `PublicIPAddress` referenced from NIC; AWS `Eip` consumed inline by `Instance.associate_public_ip` — the legacy `EipAssociation` split is widely considered an awkward anti-pattern); (b) the Timeweb dashboard places "Public IP" inside the Server-create form's "Network" section, signaling server-centric ownership; (c) the upstream bind API's `resource_type ∈ {server, balancer, database, network}` enum confirms the IP is the generic resource and the consumer is the specific one — same `floatingIPRefs` field will land on future `Balancer`, `Database`, and `Network` MR kinds. FR-016 and FR-017 below are revised accordingly; US4 (the allocate-bind flow) shifts to be Server-driven; FloatingIP-side bind state machine drops out of scope. Constitution §II compliance: bind/unbind side-effects remain single-owner (now on the Server controller).
- Q: Does the custom-resources Server path ("Произвольная" tab in the dashboard — operator types CPU/RAM/disk directly) stay deferred to feature 004, or does it expand v0.3 scope? → A: **Stays deferred to feature 004.** Concrete scope captured here so the deferral is testable: feature 004 adds (1) `Server.forProvider.resources: { location, cpu, ramMB, diskGB, diskType, bandwidthMbps, gpu?, cpuFrequencyTier?, enableLocalNetwork? }` with CEL `presetName XOR resources` (the original respec design we simplified for v0.3); (2) the resolver primitive's existing `DimServerConfigurator` dimension promoted from `fetchUnwired` stub to a real fetcher calling `GetConfigurators` (already in the generated client per T001); (3) the configurator-selection algorithm — hard-filter by `(location, diskType, enableLocalNetwork?, cpuFrequencyTier?)`, capability-filter by `requirements.{cpu, ram, disk, bandwidth}` containing the requested values, tightest-fit rank by ascending `(max_cpu, max_ramMB, max_diskGB)`, tiebreaker by lowest `configurator_id`; (4) `lockedConfiguratorID` status field (distinct from `lockedPresetID`); (5) sizing-variant-switch detection — flipping between `presetName` and `resources` on a live Server surfaces `Synced=False, reason=SizingSwitchRequiresRecreate`. Estimated 12 tasks per the original 003 phase template. Infrastructure for this path already exists in the v0.3 codebase (DimServerConfigurator registration, GetConfigurators generated method, Configurator resolver kind from feature 002).

- Q: How does a Server MR pick which Timeweb project the server lands in? → A: **Crossplane-style `projectRef` + `projectSelector` + flat `projectID` escape hatch** (mutually exclusive trio, matches the `networkRef` pattern just locked in). Refs resolve into the same-namespace `Project` MR's `status.atProvider.upstreamID`. When all three are unset the upstream account's default project is used. FR-004 amended to add the trio.
- Q: How does the operator type the OS image + version on `Server.spec.forProvider.os`? → A: **Two-field object — `image: <lowercase family slug>` + `version: <upstream version string>`** (e.g. `{ image: ubuntu, version: "24.04" }`). The resolver matches these 1:1 against `/api/v1/os/servers`'s `name` + `version`. No combined-slug shape; no flat `osID` escape hatch in v0.1 (the friendly path is the only path). FR-007 unchanged (matches the initial draft); locked here.
- Q: How does v0.1 handle public-IP behavior on Server (the dashboard's "Публичный IP +180 ₽/мес" + free IPv6 lines)? → A: **Ship a dedicated `FloatingIP` MR in this same feature** plus a `Server.forProvider.floatingIPRefs: [{name}]` list-of-refs binding mechanism. Upstream API: `/api/v1/floating-ips` for CRUD + `/{id}/bind` + `/{id}/unbind`. Spec scope expands from 2 MRs to 3 (`Server` + `Network` + `FloatingIP`). FR-016 + FR-017 + new user story US4 added below. **[SUPERSEDED 2026-06-01 (FloatingIP reversal session)]** — the original direction (FloatingIP.serverRef authoritative, Server.floatingIPRefs observe-only) was reversed mid-implementation per the "FloatingIP reference reversal + custom-server deferral" clarification above. The authoritative shape is now Server-consumes-IP. FR-016/FR-017 wording below reflects the post-reversal contract.

## User Scenarios & Testing *(mandatory)*

### User Story 1 — Provision a cloud server with a named preset (Priority: P1) 🎯 MVP

A platform operator declares a `Server` managed resource specifying: a name, a region (location), a preset slug (e.g. `premium-2-2-40-msk-1`), an OS image (e.g. `ubuntu/24.04`), and optionally one or more SSH key references. The controller resolves the preset slug + OS image to upstream IDs via the in-controller catalog resolver (the primitive shipped in feature 002), creates the server upstream, and reports `Ready=True, Synced=True` once the upstream resource reaches the active state.

**Why this priority**: This is the entire user-facing payoff of the feature. Operators get a deterministic, GitOps-driven path to a working cloud server using the sizing tier they actually pick from the dashboard — no preset IDs, no configurator IDs, no OS IDs typed by hand. SSH-key wiring through cross-resource references gives a complete "I can ssh in" experience using only Kubernetes objects, no out-of-band steps.

**Independent Test**: With the provider installed and a `ProviderConfig` carrying a valid Timeweb token, an operator applies an `SshKey` MR with a known public key, then applies a `Server` MR carrying `forProvider.presetName: <smallest-msk-1-slug>`, `forProvider.os: { image: ubuntu, version: "24.04" }`, `forProvider.location: msk-1`, and `forProvider.sshKeyRefs: [{ name: <ssh-mr-name> }]`. Within ten minutes the `Server` reports `Ready=True, Synced=True` and has `status.atProvider.upstreamID`, `status.atProvider.publicIP`, `status.atProvider.lockedPresetID`, and `status.atProvider.lockedOSID` populated. SSH login to `publicIP` with the corresponding private key succeeds. `kubectl delete server/<name>` removes the upstream resource.

**Acceptance Scenarios**:

1. **Given** a `ProviderConfig` resolving to a valid token + an `SshKey` MR in `Ready=True`, **When** the operator applies a `Server` MR with `presetName`, `location`, `os`, and `sshKeyRefs: [<sshkey-mr-name>]`, **Then** the controller resolves the slug + OS to upstream IDs from a single warm-cache catalog GET, creates the server upstream, polls until upstream state is "on", and reports `Ready=True, Synced=True` with the upstream ID + public IPv4 in `status.atProvider`.
2. **Given** a live `Server` MR in `Ready=True`, **When** the operator patches `forProvider.comment` (or `forProvider.hostname`), **Then** the controller PATCHes the upstream server, the `Synced=True` condition stays true, and `status.atProvider` reflects the new value within one reconcile cycle.
3. **Given** a live `Server` MR, **When** the operator patches `forProvider.presetName` to a different slug in the same location, **Then** the controller rejects the change with `Synced=False, reason=ImmutableFieldChange` (sizing changes require a new upstream resource per Timeweb API behavior; resize/upgrade flows land in a follow-up feature).
4. **Given** a `Server` MR with `presetName` referencing a slug that doesn't exist in the operator's catalog view, **When** the operator applies it, **Then** the MR surfaces `Synced=False, reason=ReconcileError` with the resolver's `PresetNotFound` message listing valid slugs visible to the credential (same UX as `ContainerRegistry`/`S3Bucket`).

### User Story 2 — Create a private network and attach a server to it (Priority: P2)

A platform operator declares a `Network` managed resource specifying a name, subnet CIDR (e.g. `10.10.0.0/24`), and location. They then apply a `Server` MR whose `forProvider.networkRef.name` points at that `Network` MR. The provider creates the VPC upstream, waits for it to be ready, resolves the Server's `networkRef` to the upstream VPC ID, and creates the server attached to the private network.

**Why this priority**: The dashboard exposes private networks as a first-class section in the create-server flow (image #2 in the feature description). Without this story, operators can spin up servers but can't replicate the common pattern of "multiple servers on a shared private network with internal addresses". Lower than P1 because a working public-IP-only server is independently useful, and the cross-resource reference UX is a self-contained second slice.

**Independent Test**: Apply a `Network` MR with `forProvider: { name: shared, subnetCIDR: 10.20.0.0/24, location: msk-1 }`; within one minute it reports `Ready=True, Synced=True` and `status.atProvider.upstreamID` is set. Apply a `Server` MR (per US1) with the addition of `forProvider.networkRef.name: shared`; the controller resolves the ref to the VPC's upstream ID, creates the server attached to the VPC, and the resulting server's `status.atProvider.privateIP` falls inside `10.20.0.0/24`. Apply a second `Server` against the same network; its `privateIP` is in the same CIDR and the two servers can reach each other on those addresses.

**Acceptance Scenarios**:

1. **Given** a `Network` MR in `Ready=True` with `status.atProvider.upstreamID: vpc-xyz`, **When** the operator applies a `Server` MR with `forProvider.networkRef.name: <network-mr-name>`, **Then** the resolver attaches the server to `vpc-xyz` upstream and `Server.status.atProvider.privateIP` is populated with an address inside the network's CIDR.
2. **Given** a `Network` MR whose underlying VPC is still being provisioned (`Ready=False`), **When** the operator applies a `Server` MR with a `networkRef` pointing at it, **Then** the Server reconciler waits on the Network reaching `Ready=True` before issuing the upstream Create, reporting `Synced=False, reason=Reconciling` with a message naming the dependency it is waiting on.
3. **Given** a deleted `Network` MR while a `Server` MR still references it via `networkRef`, **When** the controller next reconciles the Server, **Then** the Server surfaces `Synced=False, reason=ReconcileError` with the resolver-style "referenced Network not found" message; the upstream server itself is unaffected (the controller does not delete or detach it as a reaction to a missing CR).

### User Story 3 — Reference an externally-managed VPC by ID (Priority: P3)

A platform operator running Crossplane on a cluster shared with a dashboard-managed Timeweb account wants to launch a `Server` MR into a VPC that already exists upstream and that they do NOT want to model as a Crossplane `Network` MR (perhaps another team manages it). They set `forProvider.networkID: <vpc-id>` directly on the Server, bypassing the cross-resource reference.

**Why this priority**: Same import-friendly escape-hatch pattern as `Project` from the MVP — covers mixed dashboard/Crossplane environments without forcing every cluster to be the sole owner of every dependency. Last in priority because most operators in steady-state will be using `networkRef`; `networkID` is for migration and mixed-ownership cases.

**Independent Test**: With an existing dashboard-created VPC `vpc-abc123` already serving traffic, an operator applies a `Server` MR with `forProvider.networkID: vpc-abc123` and no `networkRef`. The controller skips the cross-resource lookup and attaches the server to that VPC directly. The Server reaches `Ready=True, Synced=True`; `status.atProvider.privateIP` falls inside the VPC's CIDR. Deleting the Server MR removes only the server upstream; the dashboard-managed VPC is untouched.

**Acceptance Scenarios**:

1. **Given** an existing upstream VPC the operator did NOT create through Crossplane, **When** the operator applies a `Server` MR with `forProvider.networkID: <upstream-vpc-id>` (and no `networkRef`/`networkSelector`), **Then** the controller attaches the server to that VPC without performing any lookup of a Kubernetes `Network` object, and `Server.status.atProvider.privateIP` reflects the assigned address inside the VPC's CIDR.
2. **Given** a `Server` MR with both `forProvider.networkRef` and `forProvider.networkID` set, **When** the operator applies it, **Then** admission rejects the resource with a CEL/validation error naming the mutually-exclusive fields.

### User Story 4 — Pin a floating IPv4 to a server (Priority: P2)

A platform operator declares a `FloatingIP` MR (pure allocation — `location` + `isDDoSGuard` only) and then a `Server` MR with `forProvider.floatingIPRefs: [{ name: <fip-mr-name> }]`. The `FloatingIP` controller allocates the IP upstream (returning the address); the `Server` controller, after the server upstream is `Ready=True`, calls `POST /floating-ips/{id}/bind` with `resource_type: "server"`. Re-pointing `floatingIPRefs` to a different IP triggers unbind + bind; clearing the trio triggers unbind only.

**Why this priority**: Same priority as US2 (private networks). The dashboard treats public IP and private network as siblings in the "Network" section of the create-server flow. Without explicit IP control, operators can't model "this server keeps the same public IPv4 across recreates" (a common GitOps requirement for DNS pinning and firewall allowlists). Lower than US1 because servers without an operator-managed floating IP still get whatever upstream-default address Timeweb assigns and can be SSHed into; this story adds the persistent-IP guarantee.

**Independent Test**: Apply a `FloatingIP` MR with `forProvider: { location: msk-1, isDDoSGuard: false }` — within one minute it reports `Ready=True, Synced=True` and `status.atProvider.ip` holds a routable IPv4. Apply a `Server` MR per US1 with the addition of `forProvider.floatingIPRefs: [{ name: <fip-mr-name> }]`. Once the Server reaches `Ready=True`, its `status.atProvider.boundFloatingIPs` lists the floating IP's upstream ID and the IP's upstream `bound_to.resource_id` matches the server's upstream ID. Patch the Server to clear `floatingIPRefs` — the Server controller calls unbind; the FloatingIP stays allocated and unbound (re-bindable). Delete the FloatingIP MR — controller deallocates.

**Acceptance Scenarios**:

1. **Given** a `FloatingIP` MR in `Ready=True` (allocated, unbound) AND a `Server` MR in `Ready=True`, **When** the operator patches the Server's `forProvider.floatingIPRefs` to include the IP's MR name, **Then** the Server controller calls `POST /floating-ips/{id}/bind` (with `resource_type: "server"`), `Server.status.atProvider.boundFloatingIPs` includes the floating IP's upstream ID within one reconcile cycle, and the bound IP is reachable from the public internet.
2. **Given** a Server bound to FloatingIP A, **When** the operator patches `floatingIPRefs` to point at FloatingIP B instead, **Then** the Server controller issues exactly one unbind (for A) + one bind (for B) upstream; bound state in both `Server.status` AND in each FloatingIP's `status.atProvider.observedBoundTo` reflects the new binding within two reconcile cycles.
3. **Given** a Server with one bound FloatingIP, **When** the operator clears `forProvider.floatingIPRefs` entirely, **Then** the Server controller calls unbind for each previously-bound IP; the FloatingIP MRs stay `Ready=True` (allocated, now unbound — re-bindable from a different Server later).
4. **Given** a Server with `floatingIPRefs.name` pointing at a FloatingIP MR that doesn't exist (or isn't yet `Ready=True`), **When** the Server reconciles, **Then** the Server surfaces `Synced=False, reason=Reconciling` with a message naming the missing/unready dependency. The Server itself is created without the binding; binding occurs on the reconcile after the FloatingIP becomes `Ready=True`.
5. **Given** a Server MR pending deletion, **When** the controller starts the delete flow, **Then** it first issues unbind for every IP in `Server.status.atProvider.boundFloatingIPs` (idempotent; 4xx tolerated for IPs already unbound), then issues server delete. Each referenced FloatingIP MR stays in the cluster (the Server doesn't own them).

### Edge Cases

- **Region not available** (Frankfurt is marked "Распродано" in the dashboard): operator types `location: fra-1` → controller forwards the request, upstream returns 4xx, MR surfaces `Synced=False, reason=ReconcileError` with the upstream message.
- **OS not available for the chosen region**: not every OS is available in every region; the controller's resolver-style enum failure surfaces `Synced=False` with the list of OS+version pairs valid for that location.
- **VPC location mismatch**: `Server.location = msk-1` but the referenced `Network.location = spb-3`. The controller surfaces `Synced=False, reason=ReconcileError` with a message naming the mismatch; the upstream API would reject the call anyway, but we catch it pre-flight to give the operator a clearer error.
- **Public IP exhausted** for the operator's account: upstream Create returns a typed quota-exceeded error; MR surfaces `Synced=False, reason=APIError` with the upstream message.
- **Concurrent SSH key revocation**: operator deletes an `SshKey` MR referenced by `sshKeyRefs` while the Server is in `Ready=True`. The controller does NOT mutate the upstream server (Timeweb's API key-list is set at create time only; remove-key flows belong to a follow-up). The `Server` MR's `Synced` condition stays true; the operator can manually clean up via the dashboard or by recreating the server.

## Requirements *(mandatory)*

### Functional Requirements

- **FR-001**: The provider MUST publish a namespaced `Server` MR kind in API group `compute.m.timeweb.crossplane.io/v1alpha1` (following the `<svc>.m.timeweb.crossplane.io` convention established in feature 001). The MR MUST be a Crossplane v2 ModernManaged resource using the shared `ProviderConfigSpec` from feature 002 (no duplicate PC plumbing).
- **FR-002**: The provider MUST publish a namespaced `Network` MR kind in API group `network.m.timeweb.crossplane.io/v1alpha1` representing a Timeweb VPC. Same v2-ModernManaged contract as `Server`.
- **FR-003**: `Server.spec.forProvider` MUST require the following operator inputs: `name` (string, max 255 chars per Timeweb), `presetName` (string slug, resolved to upstream `preset_id`), `location` (enum string of region codes — `spb-3`, `msk-1`, `nsk-1`, `ams-1`, `ala-1`, `buf-2` based on dashboard; Frankfurt `fra-1` not listed in scope since it is sold out, but the CRD MUST NOT hard-block it to allow recovery if Timeweb re-enables capacity), `os` (object with `image: <enum string>` and `version: <string>` resolved to upstream `os_id`).
- **FR-004**: `Server.spec.forProvider` MUST accept the following optional operator inputs: `hostname` (string), `comment` (string, max 255 chars), `cloud_init` (string, raw user-data script), `sshKeyRefs` (list of crossplane-style `{name}` references into the same namespace's `SshKey` MRs), `sshKeyIDs` (list of int64 upstream IDs as the escape hatch for externally-managed SSH keys, mutually exclusive with `sshKeyRefs` per-element), `networkRef` (single crossplane-style reference into a same-namespace `Network` MR), `networkSelector` (crossplane-style label selector resolving to one `Network` MR), `networkID` (string escape hatch for an externally-managed VPC), and the **project assignment trio**: `projectRef` (crossplane-style reference into a same-namespace `Project` MR), `projectSelector` (crossplane-style label selector resolving to one `Project` MR), `projectID` (int64 escape hatch for an externally-managed project). At most one of `{projectRef, projectSelector, projectID}` MAY be set per Server (admission-time CEL). If all three are unset the upstream account's default project is used. The resolved upstream project ID is recorded in `Server.status.atProvider.resolvedProjectID`.
- **FR-005**: `Server.spec.forProvider` MUST reject (admission-time CEL) the simultaneous setting of more than one of `{networkRef, networkSelector, networkID}` — at most one MAY be set per Server.
- **FR-006**: The controller MUST resolve `presetName` to an upstream server `preset_id` via the same in-controller catalog resolver primitive shipped in feature 002, using the resolver's `Preset` dimension kind. Slug rule unchanged from feature 002 (`<description_short>-<location>`, lowercase, non-`[a-z0-9-]` collapsed to `-`, optional `-<id>` disambiguator). The resolver dimension MUST be named `ServerPreset` and registered in `internal/controller/shared/resolver/dimensions.go` alongside the existing live + forward-compat dimensions.
- **FR-007**: The controller MUST resolve `os.image + os.version` to an upstream `os_id` via the resolver's `Enum` dimension kind. The resolver dimension MUST be named `ServerOSImage`; its upstream endpoint is `/api/v1/os/servers`. Operator-supplied `os.image` + `os.version` are matched against the upstream entry's `name` (lowercased) + `version`. Mismatches surface as `Synced=False, reason=ReconcileError` with a message listing the valid OS+version pairs visible in the operator's location.
- **FR-008**: The controller MUST record the resolved upstream identifiers in `Server.status.atProvider` on first successful Create: `upstreamID` (the server ID), `lockedPresetID` (resolved from `presetName`), `lockedOSID` (resolved from `os`), `publicIP` (the v4 public address if assigned), `publicIPv6` (the v6 public address if assigned, free per dashboard), `privateIP` (if a network is attached), `state` (one of the upstream server states: `on`, `off`, `installing`, `…`). Subsequent reconciles consult `lockedPresetID` + `lockedOSID` to detect drift and refuse silent re-creation.
- **FR-009**: Changing `presetName` on a `Ready=True` Server MUST surface `Synced=False, reason=ImmutableFieldChange` with a message instructing the operator to delete and recreate (Timeweb's API surface this feature targets does not include an upgrade-preset flow that is safe to expose; resize lands as a follow-up). Changing `os` likewise. Changing `hostname`/`comment` MUST PATCH the upstream server normally.
- **FR-010**: `Network.spec.forProvider` MUST require `name` (string), `subnetCIDR` (string, validated by CEL against the IPv4 CIDR pattern), `location` (same enum as Server). MUST accept optional `description` (string) and `availabilityZone` (string).
- **FR-011**: The controller MUST resolve a Server's `networkRef`/`networkSelector` via crossplane-runtime's standard reference resolver. The resolved upstream VPC ID is recorded in `Server.status.atProvider.resolvedNetworkID`. A `Server` with a `networkRef` whose target Network is not yet `Ready=True` MUST stay `Synced=False, reason=Reconciling` with a message naming the dependency.
- **FR-012**: The controller MUST detect a Server.location ↔ resolved-Network.location mismatch before issuing the upstream Create, and surface `Synced=False, reason=ReconcileError` with a clear message. Same check applies to `networkID`-imported VPCs (fetch the VPC first to read its location).
- **FR-013**: Deleting a `Server` MR (with default `managementPolicies: [*]`) MUST trigger an upstream Delete and wait for the upstream resource to disappear (idempotent on 404 per Constitution §II). Deleting a `Network` MR with active `Server` MRs referencing it via `networkRef`/`networkSelector` MUST be allowed at the Kubernetes layer (Crossplane v2 does NOT enforce cross-MR finalization), but the controller MUST log a warning per Server that loses its dependency.
- **FR-014**: The `Server` and `Network` MRs MUST publish standard Crossplane `Synced` + `Ready` conditions per Constitution §II. State mapping for `Ready`: upstream `state == on` → `Ready=True`; `installing`/`booting`/`stopped`/anything else → `Ready=False` with an appropriate reason in the message. Deletion-in-flight follows the standard runtime behavior (`Ready=False, reason=Deleting`).
- **FR-015**: The `Server` controller MUST publish a connection Secret (via `writeConnectionSecretToRef`) with at minimum `publicIP`, `privateIP` (if attached), and `hostname`. The Secret is created in the MR's namespace per Crossplane v2 modern-managed semantics.
- **FR-016**: The provider MUST publish a namespaced `FloatingIP` MR kind in API group `network.m.timeweb.crossplane.io/v1alpha1` representing a Timeweb floating IPv4 address. `spec.forProvider` MUST require `location` (same enum as `Server` / `Network`) and `isDDoSGuard` (boolean, CRD-default `false`). It SHOULD accept optional `comment` (string) and `availabilityZone` (string). The MR is **pure allocation** — it owns only `POST /api/v1/floating-ips` (Create) and `DELETE /api/v1/floating-ips/{id}` (Delete). It does **not** carry any server/balancer/database/network binding fields; the consuming MR drives bind/unbind (see FR-017). `status.atProvider` MUST carry `upstreamID`, `ip` (the assigned IPv4 address), and observable bind state read from the upstream `bound_to` field for diagnostics (`observedBoundTo: {resourceType, resourceID}` — read-only, not authoritative for reconciliation).
- **FR-017**: `Server.spec.forProvider.{floatingIPRefs, floatingIPSelector, floatingIPIDs}` is the **authoritative binding trio** for attaching floating IPs to a Server. At most one MAY be set per Server (admission-time CEL). The Server controller MUST issue `POST /api/v1/floating-ips/{id}/bind` (with `resource_type: "server"`, `resource_id: <upstreamServerID>`) for each resolved floating IP at Create time, and MUST issue `POST /api/v1/floating-ips/{id}/unbind` when the operator removes an IP from the binding trio OR deletes the Server MR (pre-delete cleanup; idempotent if upstream already unbound). The Server controller's bind/unbind calls are the SOLE authoritative source of bind-state mutation per Constitution §II (idempotent + side-effect-aware) — single-owner semantics preserved. `Server.status.atProvider.boundFloatingIPs` records the upstream IDs as observed from `GET /api/v1/servers/{id}` (an upstream observation, not from the MR refs); operators reading `kubectl describe server` see both spec (intended) and status (observed) binding state. The same `floatingIPRefs`/`floatingIPSelector`/`floatingIPIDs` field shape extends uniformly to future consumer MR kinds — `Balancer`, `Database`, and `Network` (per the upstream `bind` endpoint's `resource_type ∈ {server, balancer, database, network}` enum and the 2026-06-01 network-group commitment).

### Key Entities

- **Server** (namespaced, `compute.m.timeweb.crossplane.io/v1alpha1`): The cloud server itself. `spec.forProvider` carries the operator-facing identifiers (name, presetName, location, os, sshKeyRefs, networkRef, projectRef, **floatingIPRefs/floatingIPSelector/floatingIPIDs — authoritative binding trio per FR-017**, optional fields). `status.atProvider` carries the resolved upstream IDs + observed state + network addresses + bound floating IPs (observed from upstream). References `SshKey` MRs (existing), `Network` MRs (new), `Project` MRs (existing), `FloatingIP` MRs (new — **authoritative**, the Server controller owns bind/unbind). The Server controller is the single-owner of bind/unbind side-effects per Constitution §II.
- **Network** (namespaced, `network.m.timeweb.crossplane.io/v1alpha1`): A Timeweb VPC. `spec.forProvider` carries `name`, `subnetCIDR`, `location`, optional `description`/`availabilityZone`. `status.atProvider` carries the upstream `vpc_id`. Independently usable (no MRs forced to attach to it).
- **FloatingIP** (namespaced, `network.m.timeweb.crossplane.io/v1alpha1`): A Timeweb floating IPv4 address. **Pure allocation** post-reversal — `spec.forProvider` carries `location`, `isDDoSGuard` (default false), optional `comment`/`availabilityZone`. NO server-binding trio on this kind; the consuming Server (or future Balancer/Database/Network) MR drives binding via its own `floatingIPRefs` trio. The FloatingIP MR owns `POST /floating-ips` (Create) and `DELETE /floating-ips/{id}` (Delete) only. `status.atProvider` carries `upstreamID`, `ip`, and `observedBoundTo: {resourceType, resourceID}` read from upstream for diagnostics.
- **Resolver dimensions added**: `ServerPreset` (Preset kind, `/api/v1/presets/servers`) and `ServerOSImage` (Enum kind, `/api/v1/os/servers`). Both wired into the existing `internal/controller/shared/resolver` registry. The forward-compat `ServerConfigurator` registration from feature 002 stays at its `fetchUnwired` stub for this feature — the custom-configurator path lands later. No catalog endpoint is consulted for VPCs or floating IPs (their inputs are operator-typed scalars validated by CEL or by the upstream API itself).

## Success Criteria *(mandatory)*

### Measurable Outcomes

- **SC-001**: An operator who has applied a `ProviderConfig` + an `SshKey` can create a working Ubuntu 24.04 cloud server in `msk-1` using only Kubernetes manifests (no IDs typed by hand) and SSH into it within ten minutes from `kubectl apply` to `Ready=True`.
- **SC-002**: An operator who types an unrecognized `presetName` or an unsupported `os.image+version` pair sees a `Synced=False` condition within two reconcile cycles whose message lists the valid options visible to their credential (≥1 valid slug / OS pair must appear in the message). The operator can correct the manifest without consulting external documentation.
- **SC-003**: Two `Server` MRs created with the same `forProvider.networkRef` pointing at one `Network` MR resolve to upstream private IPs in the same VPC subnet; an operator can SSH from one to the other via private IP within 15 minutes of the second server reaching `Ready=True`.
- **SC-004**: Deleting a `Server` MR removes the upstream server within five minutes (`upstream-state observed not-found by the controller`). Deleting the corresponding `SshKey` MR before the Server has not removed the server upstream — Crossplane v2 cross-MR ownership is non-blocking, matching FR-013.
- **SC-005**: The `Server` connection Secret is created within one reconcile cycle of `Ready=True` and contains usable `publicIP` + (when networked) `privateIP` keys. Operators can wire a downstream Helm chart or Job to read this Secret without controller-side polling.
- **SC-006**: The provider's catalog GET volume against `/api/v1/presets/servers` and `/api/v1/os/servers` remains ≤1 request per `(ProviderConfig, dimension)` per TTL window even under 50 concurrent Server reconciles, verified via the existing resolver cache contract. (Inherited from feature 002 SC-004 — proves the resolver primitive carries over without code duplication.)
- **SC-007**: An operator who applies (or patches in) `Server.forProvider.floatingIPRefs: [{ name: <fip> }]` against a `Ready=True` Server sees the IP bound (upstream `bound_to` populated, `Server.status.atProvider.boundFloatingIPs` populated) within two reconcile cycles. Swapping the entry to a different `FloatingIP` MR triggers exactly one unbind + one bind upstream call; the operator can verify the new binding by SSHing to the (now-reassigned) IP within 5 minutes. Deleting the Server MR unbinds every entry in its `floatingIPRefs` before issuing the server-delete call, leaving the FloatingIP MRs allocated and re-bindable.

## Assumptions

- The dashboard's "Create Server" flow (image #1 of the user description) is the canonical operator-side reference; the dashboard's tier picker, OS picker, region picker, and network picker map 1:1 to the v0.1 `Server.spec.forProvider` fields. Anything in the dashboard that's NOT in the spec (custom configurator, dedicated CPU sub-tabs, backups, public-IP add-on toggle, software_id, avatar, DDoS guard) is **out of scope** for this feature and lands incrementally in follow-ups.
- The Timeweb API behaves as described in the vendored `docs/openapi-timeweb.json` for the in-scope endpoints (`/api/v1/servers`, `/api/v1/presets/servers`, `/api/v1/os/servers`, `/api/v2/vpcs`). Endpoint paths discovered to differ from the schema during implementation are resolved by curl-probing the live API (per `project_timeweb_underscore_envelopes` memory).
- The `terraform-provider-timeweb` (the community Terraform provider for Timeweb) is used as a structural reference for field naming and create-body shape, NOT as a code dependency. We are NOT depending on Terraform/Upjet for this provider (per `feedback_use_standard_ecosystem_tools` — the project's "no external dependency" policy applies specifically to that ecosystem).
- The shared `ProviderConfig` + `ClusterProviderConfig` pair and the in-controller resolver primitive from feature 002 (merged into `main`) are available and unchanged. This feature adds two new dimension registrations + two new MR controllers; it does NOT re-open any feature-002 contracts.
- The resolver primitive's existing dimension kinds (`Preset`, `Configurator`, `Enum`) are sufficient for this feature. `ServerPreset` → `Preset`; `ServerOSImage` → `Enum`. No new dimension kind required.
- The Crossplane v2 cross-resource reference resolver (`ResolveReferences` / `ReferenceResolverFn`) is used for `networkRef`/`networkSelector` resolution rather than a custom dual-walk. Matches the upstream-alignment direction taken in feature 002.
- This feature does NOT define resize, reboot, clone, boot-mode change, backup management, or auto-backup configuration. Those are deferred to dedicated features (each is its own Timeweb sub-API and each can ship independently of "I can create a server").
- The MVP code shipped on `main` has no external consumers for cloud-server MRs (none exist yet), so the v0.1 `Server` shape can evolve freely until a `v1beta1` is cut.
- **API group homes** (forward-compat commitment per the 2026-06-01 network-group clarification): every future network-class kind extends `network.m.timeweb.crossplane.io` — currently `Network` + `FloatingIP`; future `Router`, `Balancer`, `FirewallRule`, `SecurityGroup`. Every future VM-class kind extends `compute.m.timeweb.crossplane.io` — currently `Server`; future `Disk`, `Backup`, `Snapshot`. Adding a new kind under one of these groups is an additive evolution that does NOT require a new Go module or API group migration. Splitting into a NEW group (e.g. `loadbalancer.m.…` or `firewall.m.…`) is explicitly rejected.
