## Clarifications

### Session 2026-05-18

- Q: Managed Resource scope policy for the MVP (cluster-scoped vs. namespaced vs. both)? → A: Namespaced MRs only — every managed resource is a namespace-scoped Kubernetes object; ProviderConfig stays cluster-scoped.
- Q: Container Registry surface area for the MVP? → A: Registry + Repository + tariff-catalog data source — `ContainerRegistry` MR, `ContainerRegistryRepository` MR, and an observe-only `ContainerRegistryPreset` CRD that doubles as the worked example for User Story 3's read-only-reference design.
- Q: How should the controller respond to a change on an immutable upstream field? → A: Reject and surface — leave the upstream resource untouched, set `Synced=False` with a reason naming the immutable field, and emit a Kubernetes Event; never auto-recreate.
- Q: Localization depth for CRD field descriptions, given the API spec is Russian-only? → A: Best-effort one-line English per field, hand-written by the contributor adding the resource. No bilingual content in CRDs; no full translation of the upstream spec.
- Q: Connection Secret typing convention across published resources? → A: Typed where natural, Opaque otherwise — `ContainerRegistry` publishes a `kubernetes.io/dockerconfigjson` Secret (with both the `.dockerconfigjson` blob and raw `endpoint`/`username`/`password` keys); `K8sCluster` publishes Opaque with a `kubeconfig` key; other resources publish Opaque with documented keys.
- Q: Provider implementation approach — generate from the Terraform provider via Upjet, or hand-write native controllers? → A: **Native controllers only.** Every CRD, every API client, and every reconciler is owned by this repository. No runtime or build-time dependency on `terraform-provider-timeweb-cloud`, `crossplane/upjet`, or any other external code-generation pipeline that introduces a Terraform-provider lineage. PLAN.md §1's recommendation in favour of Upjet is hereby superseded.
- Q: MVP scope under the native-only constraint? → A: **Vertical slice.** v0.1 MVP ships ProviderConfig plus four managed resources (`Project`, `SshKey`, `S3Bucket`, `ContainerRegistry`) plus the Container Registry parent-child + data-source pair (`ContainerRegistryRepository`, `ContainerRegistryPreset`) — 7 CRDs total. The remaining 13 resources originally enumerated in PLAN.md §3.1 (Vpc, FloatingIp, Firewall, FirewallRule, K8sCluster, K8sNodeGroup, Server, ServerIp, ServerDisk, ServerDiskBackupSchedule, S3BucketSubdomain, DnsRr, NetworkDrive) move to v0.2.
- Q: Code-generation source for the native controllers? → A: **API client generated from `openapi.json` via `oapi-codegen`; CRD Go types hand-written with kubebuilder markers; `controller-gen` produces DeepCopy and CRD YAML; reconcilers hand-written.** No external runtime dependency on the generator; `oapi-codegen` runs once at build time and emits vendored Go code.
- Q: Long-term capability goal beyond MVP? → A: **Operator-needs-driven, no fixed end state.** After MVP, the next resources to implement are picked based on what adopters explicitly request. The thirteen v0.2-deferred TF resources become *candidates*, not commitments. There is no enumerated v1.0 resource list; the project does not claim "match TF" or "API parity" as a destination — those are accidents of adopter demand, not goals. The "Ultimate goal is to match timeweb-provider-terraform capabilities" wording in the original feature input is superseded by this clarification.
- Q: Tooling scope clarification — what does "no external dependency" cover? → A: **Only Terraform/Upjet code-generation lineage.** Standard Go and Kubernetes/Crossplane ecosystem tools (`golangci-lint`, `go test -cover`, `controller-gen`, `oapi-codegen`, `counterfeiter`, `kuttl`, `cosign`, etc.) are expected and to be configured rather than reinvented. The project MUST NOT ship custom in-tree replacements (e.g. a hand-rolled coverage linter) for capabilities that an established community tool already provides. Constitution III's four-case unit-test discipline is enforced by a documented test-naming convention plus PR review, not by a custom linter.
- Q: Go target version? → A: **Track the latest stable Go release.** The project's `go.mod` declares the current latest Go (today: `go 1.26.0`). When Go publishes a new release (e.g. 1.27 in six months), we bump on its release day or shortly thereafter via an explicit pull request. Dependencies that require newer Go are accepted; contributors must keep their toolchain current.
- Q: Dependency pinning posture? → A: **Float to latest.** Dependencies are not pinned by policy beyond what `go.sum` captures. Cadence is manual (`go get -u ./... && go mod tidy` on roughly a monthly rhythm). Each `go mod tidy` is a deliberate operation visible in git history; new majors land via explicit PRs.
- Q: `golangci-lint` invocation policy? → A: **Build it from source via `go run` against the project's own Go toolchain.** `hack/tools.go` imports `github.com/golangci/golangci-lint/v2/cmd/golangci-lint` so its version is pinned in `go.mod` like any other dep. The Makefile's `lint` target runs `go run github.com/golangci/golangci-lint/v2/cmd/golangci-lint run`. CI does the same. No system install of `golangci-lint` is required; the linter is always compiled against the same Go version as the project.

## User Scenarios & Testing *(mandatory)*

### User Story 1 — Provision the foundation resources declaratively (Priority: P1)

A platform operator describes the foundational pieces of a Timeweb environment — credentials, project grouping, SSH access, and object storage — as a set of Kubernetes manifests in a Git repository. After committing and applying them, every described Timeweb resource is created in the operator's Timeweb account. Subsequent edits to the manifests update the upstream resources in place; deletes remove them; drift introduced through the Timeweb dashboard is detected and reverted. This story proves the native-controller pattern across credentials handling, lifecycle CRUD with a connection Secret (S3Bucket), an immutable-bodied resource (SshKey), and a project-scoped grouping resource (Project) — establishing the shape that any deferred post-MVP resource (Vpc, K8s, Server, etc.) will follow when it is committed to.

**Why this priority**: Without this story the provider has no value. The MVP's purpose is to prove the native-controller pattern end-to-end on a small representative set, give the maintainer (and any early adopters) a working "credentials + SSH key + bucket + project" path, and unblock SC-002. The richer post-MVP resource set (networking, Kubernetes, virtual servers, DNS, network drives — see Assumptions for the candidate list) builds on the same controller scaffolding once it is validated and when adopter demand justifies it.

**Independent Test**: With the provider installed and authenticated against a Timeweb staging account, an operator applies a small manifest bundle (a ProviderConfig, a Project, an SshKey, and one S3Bucket), observes the resources appear in the Timeweb account within minutes, modifies a mutable field on S3Bucket and observes the upstream change, attempts a change on an immutable field (e.g. SshKey body) and observes the controller surface the rejection, deletes the manifests and observes the upstream resources disappear.

**Acceptance Scenarios**:

1. **Given** the provider is installed in a Kubernetes cluster and a credential Secret holding a valid Timeweb token has been referenced from a ProviderConfig, **When** the operator applies a Project manifest, **Then** the project appears in the Timeweb account and the Kubernetes resource reports both `Synced=True` and `Ready=True` within two minutes.
2. **Given** an S3Bucket resource is reconciled and ready, **When** the operator changes a mutable field on the manifest (for example bucket description or a non-destructive tag), **Then** the controller updates the upstream resource in place without recreating it and the Kubernetes resource returns to `Ready=True`.
3. **Given** an SshKey resource is reconciled and ready, **When** the operator changes a field declared immutable by the upstream API (e.g. the public key body), **Then** the upstream resource is NOT modified, the Kubernetes resource transitions to `Synced=False` with a reason naming the field, and a Kubernetes Event is emitted; the resource remains in this state until the operator reverts the change or explicitly deletes the MR.
4. **Given** a Crossplane-managed resource is deleted by the operator (`kubectl delete`), **When** the controller next reconciles, **Then** the upstream Timeweb resource is removed and the Kubernetes resource is finalized.
5. **Given** a Timeweb resource was deleted out of band through the Timeweb dashboard, **When** the controller next reconciles, **Then** the Kubernetes resource reports `Ready=False` with a clear reason and converges back to the declared spec on the following reconciliation.
6. **Given** a manifest references another managed resource (for example a `ContainerRegistry` references a `ContainerRegistryPreset` by name), **When** the referenced resource is not yet present or not yet reconciled, **Then** the dependent reconciliation waits without producing duplicate upstream resources and resumes once the reference becomes available.

### User Story 2 — Manage the Timeweb Container Registry through Crossplane (Priority: P2)

A platform operator declares a Container Registry as a Kubernetes resource. To pick a sizing tier they reference an observe-only `ContainerRegistryPreset` resource by name instead of pasting a numeric `preset_id` — the controller resolves the reference against the live `/api/v1/container-registry/presets` catalog. Once the registry is created, the operator may optionally declare `ContainerRegistryRepository` resources to manage the lifecycle (e.g. deletion) of repositories that were created implicitly by `docker push`. The registry's connection details (registry URL plus pull/push credentials) are exposed through a Kubernetes Secret so workloads can pull images without operators copying values from the Timeweb dashboard.

**Why this priority**: The Timeweb Terraform provider has no Container Registry resource — operators cannot manage it through any existing IaC tool today. It also exercises three patterns that recur across the provider: a parent-child managed resource (repository under registry), an observe-only data-source CRD that resolves a catalog ID at reconcile time, and (per the native-only architecture in Clarifications) the canonical shape of an in-repository reconciler implementation. The preset data-source acts as the worked example for User Story 3's design deliverable.

**Independent Test**: With the provider installed and authenticated, an operator lists `ContainerRegistryPreset` instances in the cluster (populated by the provider from the live catalog), applies a `ContainerRegistry` manifest that references one preset by name, verifies the registry appears in the Timeweb account, verifies a connection Secret containing the registry URL and pull credentials is written to the configured namespace, mounts the Secret into a test workload that successfully pulls an image, declares a `ContainerRegistryRepository` for one of the pushed repositories, deletes that repository through Crossplane and confirms the upstream is removed, then deletes the `ContainerRegistry` and confirms cleanup.

**Acceptance Scenarios**:

1. **Given** a valid ProviderConfig exists, **When** the provider starts, **Then** within five minutes the cluster contains one `ContainerRegistryPreset` per upstream preset returned by `/api/v1/container-registry/presets`, each with the preset's name, disk size, and price fields populated; these resources are observe-only (operator edits to `spec` are rejected or ignored).
2. **Given** the operator references an existing `ContainerRegistryPreset` by name in a `ContainerRegistry` manifest, **When** the manifest is applied, **Then** the registry is created in the Timeweb account using the resolved preset and the Kubernetes resource transitions to `Ready=True` within two minutes.
3. **Given** a `ContainerRegistry` is reconciled, **When** the operator inspects the connection Secret, **Then** it contains a registry endpoint URL plus the credentials needed to pull an image, and those credentials are not present in the resource's `spec` or `status`.
4. **Given** repositories exist under a registry (typically created implicitly by `docker push`), **When** the operator declares a `ContainerRegistryRepository` matching one of them and then deletes that resource, **Then** the upstream repository is removed and the Kubernetes resource is finalized.
5. **Given** a `ContainerRegistry` is deleted by the operator, **When** the controller next reconciles, **Then** the upstream registry is removed and the connection Secret is cleaned up. The `ContainerRegistryPreset` data-source CRDs are not affected by registry deletion.

### User Story 3 — Design deliverable on read-only references for VM creation (Priority: P3)

Many post-MVP Timeweb resources — most notably `Server` — require operators to reference identifiers that are not freely chosen, such as operating system image, hardware preset, geographical location, or software bundle. The Timeweb Terraform provider exposes these through `data.twc_*` data sources; Crossplane has no direct equivalent and instead offers several patterns (observe-only managed resources, generated catalog CRDs, manual ID lookup with documented conventions). A maintainer about to commit to one of the deferred resources can pick the project-wide approach from the design document without redoing the analysis. The `ContainerRegistryPreset` data-source CRD delivered by User Story 2 is the worked example — the document reviews it, generalizes the pattern, and recommends whether to extend the same shape to other catalog references or to switch approach.

**Why this priority**: `Server` is not in MVP, so operators do not need to face this question yet. But whenever `Server` (or another resource that depends on catalog IDs) is committed to post-MVP, the design space — multiple competing patterns, each with trade-offs — deserves a documented decision before code is written. Capturing this as a written design while the question is fresh, rather than rediscovering it later, keeps post-MVP planning unblocked. Capturing it as a deliverable here, rather than implementation, keeps MVP scope tight.

**Independent Test**: A reader of the design document, given only the document, can describe at least three candidate approaches for modeling read-only Timeweb catalog data in Crossplane, can articulate the trade-offs of each, and can identify the recommended approach with its rationale. The document references concrete Timeweb catalog endpoints (presets, OS images, locations, software, configurator) so that a future implementer can scope the work.

**Acceptance Scenarios**:

1. **Given** the design document is published under `docs/`, **When** post-MVP planning commits to a resource that depends on catalog data (e.g. Server), **Then** the maintainer can pick an approach from the document and move directly to implementation tasks without further investigation.
2. **Given** the design document, **When** an operator hits the catalog-reference problem before the corresponding resource is implemented (e.g. they need OS or preset IDs to plan their post-MVP topology), **Then** the document points them at the practical lookup steps to obtain those IDs manually.

### Edge Cases

- **Rate limiting**: The Timeweb API caps each endpoint at 20 requests per second. When dozens of resources reconcile simultaneously the provider must back off cleanly without losing requests or producing `Ready=False` on resources that are merely waiting.
- **Token rotation**: When the credential Secret referenced by a ProviderConfig is updated with a new token, in-flight and subsequent reconciliations must pick up the new value without provider restart and without leaking the old or new value to logs.
- **External-name collisions**: An operator may import a resource that was originally created through the Timeweb dashboard or Terraform. The provider must accept a manually-set external-name annotation and reconcile against the existing upstream resource without recreating it.
- **Out-of-band deletion**: Already covered in User Story 1, acceptance scenario 4. The controller must not silently recreate without first surfacing the drift through conditions.
- **Network partition**: Transient failures to reach the Timeweb API must be retried with backoff; they must not flip `Synced=False` on every poll.
- **Upstream Timeweb API introduces a breaking schema change**: When Timeweb publishes a new API revision that changes a field's type or removes a field, the Crossplane provider's release must publish a new CRD version rather than mutate the existing one, preserving the contract for existing CRs (per Constitution I). The provider's vendored `openapi.json` is bumped through an explicit pull request.
- **Secret unavailability**: When the credential Secret is missing or the ProviderConfig is misconfigured, every dependent resource must report `Synced=False` with a reason that names the missing Secret and stops calling the Timeweb API.
- **Late initialization**: Fields the operator leaves unset in their manifest but that the Timeweb API populates by default (for example, auto-generated subnet ranges) must be written back to the spec on first observation and never trigger a "drift detected" loop afterwards.

## Requirements *(mandatory)*

### Functional Requirements

- **FR-001**: The provider MUST publish exactly these Kubernetes CRDs in v0.1: `ProviderConfig` (cluster-scoped), `Project`, `SshKey`, `S3Bucket`, `ContainerRegistry`, `ContainerRegistryRepository`, and `ContainerRegistryPreset` (observe-only). The thirteen resources originally enumerated in PLAN.md §3.1 but not on this list (Vpc, FloatingIp, Firewall, FirewallRule, K8sCluster, K8sNodeGroup, Server, ServerIp, ServerDisk, ServerDiskBackupSchedule, S3BucketSubdomain, DnsRr, NetworkDrive) are deferred from this MVP and are explicitly out of scope for this feature. Post-MVP releases pick from this candidate list as adopter demand warrants (see Clarifications and Assumptions).
- **FR-002**: The provider MUST be installable in a Crossplane v2 control plane as a signed OCI package, and MUST not require manual modification of CRDs or controllers post-install.
- **FR-003**: The provider MUST authenticate to the Timeweb API using a bearer token sourced exclusively from a Kubernetes Secret referenced by `ProviderConfig.spec.credentials`. Tokens MUST NEVER appear in resource specs, status fields, log lines, Kubernetes events, or error messages.
- **FR-004**: For every published resource, the controller MUST converge the upstream Timeweb resource on the declared spec; drift introduced out of band MUST be detected on the next reconciliation cycle and either reverted or surfaced through the standard `Synced` and `Ready` conditions with a clear reason.
- **FR-005**: Every controller MUST surface reconciliation state through the standard Crossplane `Synced` and `Ready` conditions, and MUST emit structured logs through the runtime-provided logger. Ad-hoc `fmt.Println`/`log.Printf` in reconciliation paths is prohibited (per Constitution Provider Constraints).
- **FR-006**: Each `external` client method (`Observe`, `Create`, `Update`, `Delete`) MUST be safe to invoke repeatedly. `Observe` MUST be read-only. Errors from the Timeweb API MUST be classified explicitly as transient or terminal; transient errors trigger requeue, terminal errors set `Synced=False` with a reason (per Constitution Principle II).
- **FR-007**: Pull requests that add or modify any managed resource MUST include unit tests against a fake Timeweb API client covering four named sub-tests per `external` client method: `Success`, `NotFound`, `TransientError`, and `TerminalError`. Tests follow Go's standard table-driven sub-test convention (`t.Run("Success", …)`). The standard Go test runner (`go test`) is the merge gate; missing sub-tests are caught by code review against this convention, NOT by a custom in-tree linter (per Clarifications 2026-05-18). These tests MUST pass in CI before merge (per Constitution Principle III).
- **FR-008**: When any file under `apis/` is changed, the corresponding generated artifacts MUST be regenerated and committed in the same pull request. Continuous integration MUST verify the working tree is clean after running the project's code-generation target (per Constitution Principle I). Code generation in v0.1 comprises exactly two steps: `oapi-codegen` produces the typed Timeweb API client from the vendored `openapi.json` (re-run only when the OpenAPI document is bumped) and `controller-gen` produces DeepCopy methods plus CRD YAML manifests from hand-written kubebuilder-annotated Go types under `apis/`. No code-generation step in v0.1 may depend on an external Terraform provider, Upjet, or any other out-of-tree code-generation pipeline (per Clarifications 2026-05-18).
- **FR-009**: Each resource whose creation produces operator-consumable connection information MUST publish a Kubernetes Secret in the namespace the operator selects on the resource. The Secret's keys MUST be documented per resource. The Secret's **type** MUST follow the typed-where-natural convention: `ContainerRegistry` publishes a `kubernetes.io/dockerconfigjson` Secret carrying both the standard `.dockerconfigjson` blob and the extracted `endpoint`/`username`/`password` keys; `K8sCluster` publishes an Opaque Secret with a `kubeconfig` key; every other resource publishes an Opaque Secret with documented keys. When a resource produces no operator-consumable connection information, the resource MUST NOT emit a Secret.
- **FR-010**: The repository MUST contain a design document under `docs/` that surveys at least three candidate patterns for modeling Timeweb catalog data (the API endpoints traditionally consumed as Terraform `data.twc_*` sources — OS images, presets, locations, software, configurator) in Crossplane, describes the trade-offs of each (observe-only managed resources, generated catalog CRDs, manual ID lookup with documented conventions, …), references the MVP's `ContainerRegistryPreset` observe-only CRD as a worked example, and recommends a single approach for any future managed resource that needs to reference catalog data. The document MUST also describe the manual-lookup workaround that operators use today when the future resource depending on catalog data has not yet been implemented.
- **FR-011**: An example manifest MUST exist under `examples/` for every published resource, with placeholder values clearly marked so an operator can copy, edit, and apply it.
- **FR-012**: At least one continuous integration workflow MUST run the project's `reviewable` (or equivalent) target on every pull request, verifying linting, code generation cleanliness, and unit test passing. A separate release workflow MUST produce signed OCI images on each tagged release.
- **FR-013**: The provider MUST support the standard Crossplane managed-resource external-name annotation as the sole authority for upstream resource identity, allowing operators to import existing Timeweb resources without recreation (per Constitution Principle II).
- **FR-014**: Reconciliation MUST tolerate the Timeweb API's documented 20-request-per-second-per-endpoint rate limit; the provider MUST back off on HTTP 429 without escalating to terminal failures.
- **FR-015**: When the credential Secret referenced by ProviderConfig changes, subsequent reconciliations MUST pick up the new value without requiring a provider pod restart.
- **FR-016**: All managed resources MUST be published as namespace-scoped Kubernetes CRDs (Crossplane v2 namespaced MR model). ProviderConfig remains cluster-scoped; each managed resource MUST reference a ProviderConfig by name and MAY reside in any namespace authorized by the operator's RBAC. Connection Secrets are written to the same namespace as the managed resource that produced them unless the operator explicitly overrides the target.
- **FR-017**: When an operator modifies a field that the upstream Timeweb API does not permit to change after resource creation (an "immutable field"), the controller MUST leave the upstream resource untouched, MUST set `Synced=False` with a reason naming the offending field, and MUST emit a Kubernetes Event describing the rejected change. The controller MUST NOT delete and recreate the upstream resource implicitly. Each managed resource's documentation MUST list its immutable fields.
- **FR-018**: Every CRD field in the MVP MUST carry an English one-sentence description authored by the contributor adding the resource. CRD descriptions MUST NOT contain Russian text; bilingual descriptions MUST NOT be published. A field whose semantics are unclear from the upstream Russian-only API documentation MUST either receive a hand-translated one-sentence English summary or be omitted from the MVP CRD pending clarification.

### Key Entities

- **Managed Resource (MR)**: A namespace-scoped Kubernetes Custom Resource that represents exactly one upstream Timeweb cloud entity, identified by an external-name annotation. The MR carries a desired spec, an observed status, and a set of standard conditions reflecting reconciliation state. Operators isolate environments (e.g. staging vs. production) by placing MRs into separate namespaces with appropriate RBAC.
- **ProviderConfig**: A cluster-scoped resource that names a credential source (a Kubernetes Secret) and any provider-wide settings. Multiple ProviderConfigs may coexist to address multiple Timeweb tenants. Managed resources reference a ProviderConfig by name, regardless of the resource's namespace.
- **Container Registry**: A Timeweb-hosted Docker registry, identified by name, exposing a URL plus pull/push credentials. Created and deleted by the `ContainerRegistry` managed resource; sized by reference to a `ContainerRegistryPreset`.
- **Container Registry Repository**: A logical namespace within a Container Registry (created implicitly by `docker push`). The `ContainerRegistryRepository` managed resource lets operators delete or otherwise manage the lifecycle of repositories that exist under a parent `ContainerRegistry`.
- **Container Registry Preset**: An observe-only catalog entry surfaced as a Kubernetes CRD by the provider, populated from Timeweb's `/api/v1/container-registry/presets` endpoint. Holds a tariff plan's name, disk capacity, price, and any other fields the API returns. Operators reference a preset by name in `ContainerRegistry` manifests instead of pasting numeric IDs; the preset CRD itself is reconciled by the provider, not by the operator.
- **Connection Secret**: A Kubernetes Secret produced by a managed resource's controller; carries the non-sensitive (URL, name) and sensitive (credentials, key material) connection details required by downstream workloads. Typed per the convention in FR-009: `kubernetes.io/dockerconfigjson` for ContainerRegistry, Opaque (with a `kubeconfig` key) for K8sCluster, Opaque otherwise.
- **Design Document (read-only references)**: A Markdown deliverable under `docs/` that records the chosen approach for handling Timeweb catalog data (OS images, presets, locations, software, configurator) in future provider versions.

## Success Criteria *(mandatory)*

### Measurable Outcomes

- **SC-001**: An operator can provision the v0.1 MVP foundation (project, SSH key, S3 bucket, container registry with at least one repository) entirely through declarative manifests applied to a Kubernetes cluster running the provider, with zero manual steps in the Timeweb dashboard required for the resources covered by the MVP.
- **SC-002**: At least three distinct resource kinds — Project, SshKey, and S3Bucket — are round-tripped (create → modify → delete) against a real Timeweb staging account before the MVP is declared done. These three are chosen because together they exercise the credential path, the connection-Secret publishing path, and a mutable-field update without incurring the cost or complexity of standing up a Kubernetes cluster or a virtual server. The remaining MVP resources are validated only by CRD installation and by applying example manifests; resource-specific runtime bugs are accepted as a known-unknown for the v0.1 release and tracked as follow-up issues.
- **SC-003**: A first-time contributor with a working Go toolchain can clone the repository, run the project's documented bootstrap command, and produce a buildable provider binary plus regenerated CRDs in under thirty minutes.
- **SC-004**: Every tagged release of the provider publishes a signed OCI image consumable by a Crossplane v2 control plane; install instructions in `README.md` reflect the latest release.
- **SC-005**: An operator who applies a Container Registry manifest can, within ten minutes and without consulting the Timeweb dashboard, reference the produced `kubernetes.io/dockerconfigjson` Secret as a workload's `imagePullSecrets` entry and successfully pull a private image from the new registry — no manual Secret transformation step required.
- **SC-006**: A reader of the read-only-references design document can identify the document's recommended approach for handling Timeweb catalog data, the at-least-three considered alternatives, and the trade-offs of each, without consulting the document's author.
- **SC-007**: Every published managed resource has at least one example manifest under `examples/` that an operator can copy, fill in three or fewer placeholder values, and apply successfully against their own Timeweb account.
- **SC-008**: Continuous integration on every pull request rejects any change under `apis/` that fails to regenerate dependent artifacts, holding the constitution's CRD Contract Stability principle without manual reviewer effort.

## Assumptions

- The vendored Timeweb OpenAPI document (`docs/openapi-timeweb.json` after relocation, currently at `openapi.json`) is the single source of truth for the Timeweb API surface. It is updated through explicit pull requests; no automated re-vendoring runs in CI.
- Tooling boundary clarification (per Clarifications 2026-05-18): the project uses **standard Go and Kubernetes/Crossplane ecosystem tools** as build-time and CI utilities — `golangci-lint`, `go test`, `go tool cover`, `controller-gen`, `oapi-codegen`, `counterfeiter`, `kuttl`, `cosign` — and configures them rather than reinventing them. The earlier "no external dependency" directive constrains code-generation **lineage** (no Terraform/Upjet provider source); it does not constrain build-time tooling.
- Resources deferred from MVP (Vpc, FloatingIp, Firewall, FirewallRule, K8sCluster, K8sNodeGroup, Server, ServerIp, ServerDisk, ServerDiskBackupSchedule, S3BucketSubdomain, DnsRr, NetworkDrive) are *candidates* for post-MVP work, not commitments — per Clarifications 2026-05-18, the post-MVP roadmap is operator-needs-driven and the next resources to ship are picked based on what adopters explicitly request. The native-controller pattern proven by the MVP — credentials, lifecycle, immutable-field handling, connection-Secret typing, parent-child references, observe-only data sources — is the template each post-MVP resource will follow when it is committed to.
- "Scaffolding" in the user's request includes the repository skeleton, the credential-handling provider configuration, the in-repository code-generation pipeline (CRD types + DeepCopy + Timeweb API client from `openapi.json`), continuous integration, and the release pipeline. It does not require automated end-to-end tests against a real Timeweb account for every resource; live-account validation is bounded by SC-002.
- The design deliverable in User Story 3 is a Markdown document under `docs/`, not a code implementation. The user described it as "research".
- Distribution targets `ghcr.io/lebedevdsl/...` (the maintainer's personal GitHub Container Registry namespace). A public listing on a Crossplane provider catalog is deferred to a future feature.
- The Container Registry surface in the MVP comprises three CRDs: a `ContainerRegistry` MR with full lifecycle (create, observe, update for description/tariff, delete) plus connection-Secret publishing; a `ContainerRegistryRepository` MR for repository deletion under a parent registry; and a `ContainerRegistryPreset` observe-only data source CRD that exposes the tariff catalog. Per-image access tokens, retention policy management, and other advanced features remain out of scope and may be added later.
- Operators authoring `Server` (VM) manifests for the MVP look up the required OS image, preset, and location identifiers manually in the Timeweb dashboard or via direct API calls. The User Story 3 design document records the manual procedure and proposes the long-term replacement.
- The Constitution at `.specify/memory/constitution.md` (v1.0.0) is the binding standard for CRD stability, idempotent reconciliation, and controller test discipline. Functional requirements in this spec reference its principles where applicable.
- The provider targets Crossplane v2 from day one with the native namespaced managed-resource model (per Clarifications 2026-05-18). Behavior on Crossplane v1 control planes is undefined and out of scope.
- Russian-only API descriptions in the Timeweb OpenAPI document are NOT translated wholesale. Per the Clarifications session, every CRD field carries an English one-line description hand-authored against the MVP resource set (see FR-018). The upstream Russian spec remains the source of truth for behavior; CRD descriptions are operator-facing summaries.
