# v0.9.0

Stabilization round. One **breaking (alpha)** change to the `Cdn` kind (see
below); everything else is non-breaking. No connection-secret key changes.

## Fixed

- **`Cdn` status-write wedge (reconcile + delete freeze).** The read-only
  `status.atProvider.observedSettings` mirror reused the input `cache` struct,
  and the settings mirror emitted **both** `queryStringInCacheKey` and
  `queryStringCacheKeyMode` from one upstream `query_args` — tripping the
  mutual-exclusion CEL on every status write (`cannot update managed resource
  status … mutually exclusive`). With status writes rejected, the resource
  could neither converge nor be deleted (the finalizer never cleared). The
  redundant `queryStringInCacheKey` bool is **removed** (see Breaking); the
  mirror now reflects the single `queryStringCacheKeyMode` setting.

- **Shared rate budget.** The Timeweb API client (rate limiter + HTTP transport)
  was built **per reconcile**, so concurrent reconciles each got their own 2 r/s
  budget — the aggregate rate scaled with fleet size and tripped Timeweb's 429 /
  egress ban (which could freeze a resource's status mirror). The limiter and
  transport are now **process-global, per host**, so total egress respects one
  budget regardless of how many resources reconcile at once. Per-request bearer
  auth stays isolated across ProviderConfigs.
- **S3User credential integrity (data-loss fix).** The connection Secret was
  re-published on every Observe/Update, but the upstream user GET omits the secret
  key — so steady-state reconciliation could overwrite a published `secret_key`
  with a blank value, breaking every consumer irrecoverably. Credentials are now
  published **only from Create**; Observe/Update leave the Secret untouched. An
  adopted user whose secret key can't be retrieved surfaces a clear condition
  instead of a blank Secret. The Secret's `region` is now derived from the primary
  granted bucket (was implied by a hardcoded endpoint); `region` is added as an
  additive key.
  - **Behavior note:** because credentials are now published only from Create, a
    manually-deleted S3User connection Secret is no longer auto-rebuilt on the
    next reconcile (the upstream user GET can't return the secret key). Recover by
    deleting and recreating the `S3User`.
- **Uniform capped backoff.** `SetupNetwork`, `SetupFloatingIP`, and `SetupCluster`
  lacked the 60s-capped reconcile-error backoff every other controller uses — added,
  so no kind can hammer the upstream on the runtime default's fast early cadence.

## Breaking (alpha)

- **`Cdn.spec.forProvider.cache.queryStringInCacheKey` removed.** It was
  redundant with `queryStringCacheKeyMode: all` (both mapped to upstream
  `query_args: {mode: "all"}`). Replace `queryStringInCacheKey: true` with
  `queryStringCacheKeyMode: all`; drop `queryStringInCacheKey: false` (omitting
  the mode disables query-string caching).

## Changed

- De-duplicated the account-admin-key derivation into one shared implementation
  (the two copies had drifted; the surviving one classifies network errors and
  checks the secret key). Behaviour-preserving except that fix.
- New `docs/conditions.md` — every condition reason the controllers emit, with
  remediation, including the runtime gotcha where a terminal reason surfaces as
  `ReconcileError` (visible only in Events).
