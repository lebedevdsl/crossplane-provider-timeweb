# v0.4.0

## Added

- **`S3User` kind** (`objectstorage.m.timeweb.crossplane.io/v1alpha1`) — scoped,
  least-privilege object-storage credentials, replacing the account-admin keys
  `S3Bucket` used to hand out.
  - **Grants:** `spec.forProvider.bucketAccess[]` — `read` / `read-write` / `admin`
    per bucket, by `bucketRef` (an `S3Bucket` in the same namespace) or `bucketName`.
  - **Connection Secret:** scoped `access_key` / `secret_key` / `endpoint` / `bucket`.
  - **Policy:** all grants render to one merged inline policy (`iam-user-policy`)
    matching the Timeweb panel, with semantic (Sid- and order-insensitive) drift
    detection.
  - **Protocols:** identity via Timeweb `/api/v2/storages/users` REST; grants via the
    AWS IAM Query API (SigV4) against `panel.s3.twcstorage.ru`.
  - **Signing keys:** the account super-user S3 keys are derived at runtime, never
    cached.
  - **Lifecycle:** `name` is immutable; level changes apply in place without rotating
    the issued keys.
- **`S3Bucket.status.attachedUsers`** (+ `ATTACHED` print column) — read-only
  bucket-side mirror of which users hold a grant and at what level. Observational
  only; `S3User` stays the sole writer of grants.

## Changed

- **Breaking (alpha): `S3Bucket` connection Secret no longer emits
  `access_key` / `secret_key`.**
  - These were account-admin keys — full access to every bucket and the storage API.
  - The Secret keeps `endpoint` / `bucket` / `region`.
  - **Migrate:** create an `S3User` and consume its connection Secret (see `README.md`
    → "Object-storage credentials").

## Fixed

- **`S3User` deletion no longer deadlocks** when its referenced `S3Bucket` is deleted
  first. The `Connect`-time `bucketRef` gate is relaxed during deletion, so the
  finalizer clears even when the target bucket is already gone (the identity and
  policy still delete by external-name).
