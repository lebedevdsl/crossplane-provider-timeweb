# v0.4.0

## Added

- **`S3User` kind** (`objectstorage.m.timeweb.crossplane.io/v1alpha1`) — scoped,
  least-privilege object-storage credentials, replacing the account-admin keys
  `S3Bucket` used to hand out. `spec.forProvider.bucketAccess[]` grants
  `read` / `read-write` / `admin` per bucket (`bucketRef` to an `S3Bucket` in the
  same namespace, or `bucketName`); the connection Secret carries the scoped
  `access_key` / `secret_key` / `endpoint` / `bucket`. All grants render to one
  merged inline policy (`iam-user-policy`) matching the Timeweb panel, with
  semantic (Sid- and order-insensitive) drift detection. Identity is created via
  Timeweb's `/api/v2/storages/users` REST; grants are applied via the AWS IAM
  Query API (SigV4) against `panel.s3.twcstorage.ru`. The account super-user S3
  keys that sign IAM calls are derived at runtime and never cached. `name` is
  immutable; level changes apply in place without rotating the issued keys.
- **`S3Bucket.status.attachedUsers`** (+ `ATTACHED` print column) — read-only
  bucket-side mirror of the users granted access and their level. Observational
  only; `S3User` remains the sole writer of grants.

## Changed

- **Breaking (alpha): `S3Bucket` connection Secret no longer emits
  `access_key` / `secret_key`.** Those were account-admin keys with full access to
  every bucket and the storage API. The Secret keeps `endpoint` / `bucket` /
  `region`. Consumers that read bucket credentials must migrate to an `S3User`
  and consume its connection Secret (see `README.md` → "Object-storage
  credentials").

## Fixed

- **`S3User` deletion no longer deadlocks** when its referenced `S3Bucket` is
  deleted first. The `Connect`-time `bucketRef` gate is now relaxed during
  deletion, so the finalizer is removed even when the target bucket is already
  gone (the user identity and policy still delete by external-name).
